# Паттерны микросервисной архитектуры — подготовка к собеседованию

Подробное описание паттернов микросервисов для подготовки к техническим собеседованиям (Java Backend).

---

# Введение в паттерны микросервисов

## Зачем нужны паттерны в распределённых системах

- Распределённые системы сталкиваются с **типовыми проблемами**: отказы узлов, задержки сети, согласованность данных, обнаружение сервисов.
- **Паттерны** — проверенные способы решать эти проблемы единообразно и предсказуемо.
- Они дают общий язык команде и уменьшают риск «изобретения велосипеда» с ошибками.

## Почему микросервисам нужны особые архитектурные паттерны

- Микросервисы — это **много независимых процессов**, связанных сетью. Классические паттерны одного процесса (монолит) здесь не подходят.
- Нужны паттерны для: **взаимодействия между сервисами**, **управления данными без общей БД**, **устойчивости к сбоям**, **постепенной миграции** и т.д.

## Типичные вызовы в микросервисных системах

| Вызов | Суть проблемы |
|-------|----------------|
| **Взаимодействие сервисов** | Синхронные вызовы создают связность и каскадные сбои; нужны шлюзы, очереди, контракты. |
| **Согласованность данных** | Нет единой транзакции между сервисами; нужны Saga, eventual consistency, события. |
| **Устойчивость к сбоям** | Один упавший сервис не должен ронять остальные; нужны retry, timeout, circuit breaker, bulkhead. |
| **Обнаружение сервисов** | Адреса инстансов динамичны; нужен реестр (service discovery). |
| **Масштабирование** | Нужно масштабировать выборочно и не перегружать один сервис; балансировка, ограничение нагрузки. |

# Паттерн API Gateway

## Что это и зачем

**API Gateway** — единая точка входа для клиентов. Все запросы идут на шлюз; шлюз маршрутизирует их к нужным микросервисам и выполняет сквозные функции (аутентификация, лимиты, агрегация).

**Зачем в микросервисах:** клиенту не нужно знать десятки URL и сервисов; внутренняя топология скрыта; аутентификация, лимиты и логирование делаются в одном месте.

## Обязанности

| Обязанность | Описание |
|-------------|----------|
| **Маршрутизация** | По пути, заголовкам, методу — перенаправление на соответствующий сервис. |
| **Аутентификация** | Проверка JWT/OAuth2, выдача или проверка токенов на границе. |
| **Rate limiting** | Ограничение частоты запросов по клиенту/ключу, защита от перегрузки. |
| **Агрегация запросов** | Один запрос клиента → несколько вызовов к сервисам → объединённый ответ (BFF). |

Дополнительно: терминация SSL, сжатие, кэширование, логирование.

## Схема

```
  Клиенты (Web, Mobile, Partners)
           │
           ▼
  ┌─────────────────────────────────┐
  │         API Gateway             │
  │  routing · auth · rate limit    │
  └─────────────┬───────────────────┘
                │
    ┌───────────┼───────────┬──────────────┐
    ▼           ▼           ▼              ▼
┌───────┐  ┌───────┐  ┌───────┐  ┌──────────┐
│Order  │  │User   │  │Product│  │Payment   │
│Service│  │Service│  │Service│  │Service   │
└───────┘  └───────┘  └───────┘  └──────────┘
```

## Примеры

- **Spring Cloud Gateway** — маршруты по предикатам и фильтрам, интеграция с Spring Security, реактивный стек.
- **Kong** — шлюз на OpenResty, плагины для аутентификации, rate limiting, логирования.
- **NGINX** — reverse proxy, балансировка, кэш; часто как входной прокси или простой шлюз.

## Когда использовать

- Есть несколько клиентов (веб, мобильные, партнёры) и множество микросервисов.
- Нужна единая точка для аутентификации, лимитов и маршрутизации.
- Нужно скрыть внутреннюю структуру и агрегировать ответы (BFF).

# Паттерн Service Discovery

## Почему нельзя использовать фиксированные IP

В облаке и Kubernetes инстансы сервисов **создаются и уничтожаются** динамически, меняются IP и порты. Жёстко прописанные адреса в конфигурации делают систему хрупкой и не позволяют масштабировать и перезапускать сервисы без смены конфигурации.

## Динамическая регистрация и реестр

- При **старте** сервис регистрируется в **реестре** (service registry): имя сервиса + адрес (host:port) + метаданные.
- Периодически отправляет **heartbeat**; при остановке снимается с регистрации (или истекает TTL).
- **Клиент** или **прокси** запрашивает у реестра список инстансов по имени и выбирает один для вызова (round-robin, random, по зоне).

## Два подхода

| Подход | Описание |
|--------|----------|
| **Client-side discovery** | Клиент (или SDK в сервисе) сам обращается к реестру, получает список инстансов и выбирает один. Пример: Eureka + Ribbon/LoadBalancer в Spring. |
| **Server-side discovery** | Клиент обращается к балансировщику/прокси по фиксированному имени; балансировщик обращается к реестру и направляет запрос на инстанс. Пример: Kubernetes Service + kube-proxy. |

## Схема (регистрация и обнаружение)

```
  ┌─────────────┐                    ┌──────────────────┐
  │  Service A  │── register ───────►│  Service Registry │
  └─────────────┘                    │  (Eureka, Consul) │
  ┌─────────────┐                    └────────▲──────────┘
  │  Service B  │── register ───────►         │
  └─────────────┘                             │ lookup
  ┌─────────────┐                    ┌────────┴──────────┐
  │  Service C  │── lookup ──────────│  Client / Gateway │
  │  (client)   │◄── list of A,B ────│  (get instances)  │
  └─────────────┘                    └───────────────────┘
```

## Примеры

- **Netflix Eureka** — реестр с REST API; Spring Cloud интегрирует регистрацию при старте и клиентский выбор инстанса (LoadBalancer, OpenFeign).
- **Consul** — реестр + health checks, KV, multi-datacenter; Spring Cloud Consul.
- **Kubernetes** — встроенный discovery: у каждого Service есть DNS-имя (например, `http://order-service`), поды автоматически участвуют в сервисе.

# Паттерн Database per Service

## Суть

У **каждого микросервиса своя БД** (или своя схема/набор таблиц). Другие сервисы **не обращаются к этой БД напрямую** — только через API или события. Данными владеет только один сервис.

## Децентрализованное владение данными

- Сервис — единственный источник правды для своей доменной области.
- Изменение схемы не требует согласования с другими сервисами; деплой сервиса и миграции его БД независимы.

## Почему общая БД нежелательна

- **Жёсткая связность** — изменение схемы затрагивает несколько команд и сервисов.
- **Единая точка отказа** — падение БД роняет всё.
- **Невозможность независимого масштабирования и деплоя** по сервисам.
- Нарушается граница владения данными и контекстов (DDD).

## Плюсы и минусы

| Плюсы | Минусы |
|-------|--------|
| Независимый деплой и миграции схемы | Нет распределённых ACID-транзакций |
| Выбор подходящей БД под задачу | Согласованность только через API/события (eventual consistency) |
| Изоляция отказов по данным | Дублирование данных и сложность консистентности |
| Масштабирование БД по сервисам | Сложнее запросы, затрагивающие несколько доменов (нужна композиция или CQRS) |

## Проблема согласованности

Данные, которые нужны нескольким сервисам, синхронизируются через **события** или **вызовы API**, а не через общую транзакцию. Это приводит к **eventual consistency** и необходимости паттернов вроде **Saga** для бизнес-транзакций, охватывающих несколько сервисов.

# Паттерн Saga

## Проблема распределённых транзакций

Операция пользователя может затрагивать **несколько сервисов** (например, заказ → списание товара → оплата). В монолите это одна транзакция; в микросервисах у каждого сервиса своя БД — **единой 2PC-транзакции нет**.

## Почему 2PC в микросервисах почти не используют

- **2PC (two-phase commit)** требует координатора и блокировок; при падении узлов и сети это даёт блокировки и сложность восстановления.
- Микросервисы стремятся к слабой связности и отказоустойчивости; 2PC усложняет архитектуру и создаёт общие точки отказа.

## Два подхода к Saga

### Choreography (хореография)

- Нет центрального оркестратора. Каждый сервис выполняет **свою локальную транзакцию** и публикует **событие**. Другие сервисы подписаны на события и выполняют свои шаги (или компенсации).
- Плюсы: слабая связность, нет единой точки отказа. Минусы: сложно отслеживать поток, трудно добавлять шаги.

### Orchestration (оркестрация)

- **Центральный оркестратор** (отдельный сервис или процесс) вызывает сервисы по шагам и при ошибке запускает **компенсирующие транзакции** в обратном порядке.
- Плюсы: явный поток, проще мониторинг и отладка. Минусы: оркестратор становится критичным компонентом.

## Пример: заказ, оплата, склад

- **Order Service** — создаёт заказ, резервирует (или отменяет при отказе).
- **Payment Service** — списывает деньги (или возвращает при компенсации).
- **Inventory Service** — резервирует товар (или снимает резерв при компенсации).

При сбое на любом шаге выполняются **компенсирующие транзакции** (отмена резерва, возврат платежа, отмена заказа).

## Компенсирующие транзакции

- **Компенсация** — действие, семантически обратное уже выполненному (откат в бизнес-смысле, а не откат БД другого сервиса).
- Saga не даёт ACID: возможны временная несогласованность и дублирующие компенсации; нужна идемпотентность.

## Схема (оркестрация)

```
  Orchestrator
       │
       ├──► Order Service: CreateOrder ──► OK
       │
       ├──► Payment Service: ReservePayment ──► OK
       │
       ├──► Inventory Service: ReserveItem ──► FAIL
       │
       └──► Compensate: RefundPayment, CancelOrder
```

# Паттерн Circuit Breaker

## Каскадные сбои

Если сервис B перестаёт отвечать или отвечает очень медленно, все вызывающие его сервисы начинают **копировать ресурсы в ожидании** (потоки, соединения). В итоге падает не только B, но и вызывающие — это **каскадный сбой**.

## Зачем circuit breaker

**Circuit Breaker** ограничивает вызовы к сбойному сервису: после серии ошибок «размыкает цепь» и какое-то время **не вызывает** сервис (сразу возвращает ошибку или fallback), давая ему восстановиться. Периодически проверяет доступность (half-open) и при успехе снова разрешает вызовы.

## Состояния

| Состояние | Поведение |
|-----------|------------|
| **Closed** | Вызовы проходят к сервису. При достижении порога ошибок переход в Open. |
| **Open** | Вызовы **не** идут к сервису; сразу возвращается ошибка или fallback. После таймаута переход в Half-Open. |
| **Half-Open** | Ограниченное число пробных вызовов. При успехе — Closed, при ошибке — снова Open. |

## Примеры

- **Resilience4j** — библиотека для Java (Spring Boot): Circuit Breaker, Retry, RateLimiter, Bulkhead. Рекомендуется в новых проектах.
- **Hystrix** (Netflix) — исторически использовался в Spring Cloud; разработка остановлена, заменяется Resilience4j или платформенными средствами.

## Когда использовать

- Синхронные вызовы к внешним или внутренним сервисам, которые могут быть перегружены или недоступны.
- Чтобы не накапливать зависшие запросы и не распространять сбой на вызывающую сторону.

# Паттерн Retry

## Временные сбои

Часть сбоев **временные** (transient): таймаут сети, перегрузка сервиса, кратковременная недоступность БД). Повтор запроса через некоторое время часто приводит к успеху.

## Автоматические повторы

**Retry** — автоматический повтор запроса при определённых ошибках (например, таймаут, 503). Ограничения: **максимальное число попыток** и **не повторять при идемпотентности** (например, не повторять POST создания заказа без идемпотентного ключа).

## Экспоненциальная задержка (exponential backoff)

Увеличивать паузу между попытками (например, 1s → 2s → 4s), чтобы не усугублять перегрузку целевого сервиса и не создавать «шторм» повторов. Часто комбинируется с jitter (случайная добавка), чтобы разнести повторы от разных клиентов.

## Реализация

- Ограничить число попыток и типы ошибок для retry (не повторять на 4xx клиента, кроме 429).
- Учитывать идемпотентность операции; при неидемпотентных операциях — только безопасные повторы или идемпотентные ключи на стороне сервиса.
- В Java: Resilience4j Retry, Spring Retry, ручная реализация с backoff.

# Паттерн Timeout

## Зачем таймауты в распределённых системах

Без таймаута вызов к «зависшему» сервису может **бесконечно держать поток и соединение**. Накопление таких вызовов исчерпывает пулы потоков и соединений и приводит к падению вызывающего сервиса.

## Предотвращение долгих блокировок

- Для **каждого** внешнего вызова (HTTP, gRPC, БД, очередь) задаётся **максимальное время ожидания**.
- По истечении времени вызов прерывается (и при необходимости повторяется по политике retry). Ресурсы освобождаются.

## Типичные стратегии

- **Короткие таймауты** для быстрых операций; более длинные — только там, где это обосновано (долгие отчёты, экспорт).
- **Иерархия таймаутов**: внешний клиент > шлюз > внутренний сервис; внутренний таймаут должен быть меньше внешнего, чтобы успеть вернуть ответ или fallback.
- Настройка на уровне клиента (Feign, RestTemplate, WebClient, драйвер БД).

# Паттерн Bulkhead

## Изоляция ресурсов

**Bulkhead** (переборка) — изоляция ресурсов (потоки, соединения) по **назначению** или по **зависимости**. Цель: сбой или перегрузка одного «отсека» не исчерпывают все ресурсы процесса и не роняют остальные операции.

## Защита от потребления всех ресурсов одним сервисом

Если один удалённый сервис тормозит и занимает все потоки из общего пула, другие вызовы (к здоровым сервисам) тоже перестают выполняться. **Отдельные пулы** для разных зависимостей ограничивают ущерб: падает только тот «отсек», который обращается к сбойному сервису.

## Примеры

- **Отдельные пулы потоков** — для вызовов к сервису A и к сервису B разные ExecutorService с ограниченным размером; переполнение одного не блокирует другой.
- **Отдельные пулы соединений** — например, отдельный connection pool к одной БД или к одному внешнему API; перегрузка одного пула не забирает соединения у других.

В Java: Resilience4j Bulkhead (ограничение параллельных вызовов), настройка отдельных пулов в RestTemplate/WebClient/Feign, отдельные DataSource для разных БД.

# Паттерн Strangler Fig (Strangler)

## Постепенная миграция монолита в микросервисы

Вместо «большого взрыва» переписывания монолита **по частям заменяют** его функциональность микросервисами. Старый монолит продолжает работать; новый трафик по мере готовности идёт на новые сервисы. Со временем монолит «обвит» новыми сервисами и в итоге может быть отключён.

## Как это работает

- Перед монолитом ставится **шлюз или прокси** (или маршрутизация по URL/заголовкам).
- Запросы к **новым** возможностям направляются в микросервисы; остальные — в монолит.
- По мере переноса функций маршруты переключают на микросервисы; код в монолите по этим функциям со временем удаляют или отключают.

## Пример миграции

1. Поднять API Gateway; по умолчанию маршрутизация в монолит.
2. Вынести «Каталог товаров» в отдельный сервис; настроить маршрут `/catalog/**` на новый сервис.
3. Вынести «Заказы»; маршрут `/orders/**` на order-service; при необходимости монолит вызывает order-service по API.
4. Постепенно переносить остальные модули; в конце отключить перенесённые части монолита.

# Паттерн Event-Driven Architecture

## Асинхронное взаимодействие через события

Сервисы не вызывают друг друга напрямую по запросу; **производитель** публикует **событие** (что произошло); **потребители** подписаны на типы событий и обрабатывают их асинхронно. Взаимодействие идёт через **брокер сообщений** или **событийную шину**.

## Производители и потребители

- **Producer** — сервис, который при изменении состояния публикует событие (например, «Заказ создан», «Платёж проведён»).
- **Consumer** — сервис, подписанный на события и выполняющий свою логику (обновление read-модели, отправка уведомления, запуск следующего шага Saga).

## Примеры технологий

- **Apache Kafka** — распределённый лог событий, высокая пропускная способность, хранение истории; подходит для стриминга и event sourcing.
- **RabbitMQ** — брокер сообщений с очередями и обменниками; гибкая маршрутизация, проще для классических очередей задач.

## Плюсы

- **Слабая связность** — производитель не знает потребителей; добавление нового потребителя не меняет производителя.
- Буферизация нагрузки, сглаживание пиков.
- Возможность воспроизведения и повторной обработки (особенно с Kafka).

# Паттерн CQRS

## Разделение команд и запросов

**CQRS** (Command Query Responsibility Segregation) — разделение модели на **запись** (commands, write model) и **чтение** (queries, read model). Запись и чтение могут использовать разные хранилища, схемы и даже технологии.

## Отдельные модели чтения и записи

- **Write model** — оптимизирована для валидации и применения изменений (часто одна агрегатная модель, событие в лог или запись в БД).
- **Read model** — оптимизирована для запросов (денормализованные проекции, индексы, кэш). Обновляется асинхронно (из событий или из write-модели).

## Выигрыш по масштабированию

- Чтение и запись можно масштабировать **независимо** (больше реплик для чтения).
- Можно выбрать подходящее хранилище для чтения (например, Elasticsearch для поиска, реляционная БД для отчётов).

## Когда полезен

- Высокая нагрузка на чтение, сложные запросы и отчёты.
- Разные требования к консистентности для чтения и записи (допустима eventual consistency для read-моделей).
- Сложная доменная модель записи и упрощённые проекции для чтения.

# Паттерн Event Sourcing

## Хранение событий вместо текущего состояния

**Event Sourcing** — вместо хранения только текущего «снимка» сущности хранятся **все события** (факты), которые привели к этому состоянию. Текущее состояние получают **применением** последовательности событий к пустой модели (replay).

## Восстановление состояния из событий

- **Состояние** = начальное состояние + событие₁ + событие₂ + … + событиеₙ.
- При необходимости можно пересобрать состояние на любую точку времени, построить новые read-модели, отладить историю.

## Плюсы и сложность

| Плюсы | Сложности |
|-------|-----------|
| Полная история изменений, аудит | Сложнее запросы «текущего» состояния (нужны снапшоты или проекции) |
| Возможность новых проекций и read-моделей из старых событий | Версионирование событий и миграции схемы событий |
| Естественная интеграция с event-driven и CQRS | Рост хранилища событий, необходимость снапшотов и TTL/архивации |

# Шпаргалка для собеседования

Краткие формулировки, которыми можно пользоваться на техническом интервью.

---

**Что такое паттерн API Gateway?**  
Единая точка входа для клиентов. Шлюз маршрутизирует запросы к микросервисам, выполняет аутентификацию, rate limiting и при необходимости агрегирует ответы. Скрывает внутреннюю топологию и упрощает клиентский код.

---

**Что такое Service Discovery и зачем он нужен?**  
В облаке инстансы сервисов меняют IP и порты. Service Discovery — это реестр, в котором сервисы регистрируются при старте, а клиенты или шлюз по имени сервиса получают список инстансов для вызова. Примеры: Eureka, Consul, Kubernetes.

---

**В чём суть паттерна Database per Service?**  
У каждого микросервиса своя БД; другие сервисы не обращаются к ней напрямую, только через API или события. Это даёт независимый деплой и владение данными, но убирает распределённые ACID-транзакции и требует согласованности через Saga или eventual consistency.

---

**Что такое Saga и какую проблему она решает?**  
Saga решает проблему бизнес-операций, затрагивающих несколько сервисов со своими БД. Вместо одной распределённой транзакции выполняется последовательность локальных шагов; при сбое запускаются компенсирующие транзакции (откат в бизнес-смысле). Бывает хореография (события между сервисами) и оркестрация (центральный оркестратор).

---

**Какую проблему решает Circuit Breaker?**  
Защита от каскадных сбоев: когда зависимый сервис перестаёт отвечать, вызовы не должны бесконечно ждать и исчерпывать ресурсы вызывающего. Circuit breaker после серии ошибок «размыкает цепь» и временно не вызывает сбойный сервис, возвращая ошибку или fallback и давая ему восстановиться.

---

**Зачем нужны retry и timeout в микросервисах?**  
Retry — для обработки временных сбоев (сеть, перегрузка); повтор с ограничением попыток и backoff. Timeout — чтобы не держать поток и соединение бесконечно при «зависшем» сервисе и не исчерпывать ресурсы; после таймаута вызов прерывается.

---

**Что такое паттерн Bulkhead?**  
Изоляция ресурсов (потоки, соединения) по разным зависимостям или по назначению. Цель — чтобы сбой или перегрузка одного сервиса не забирала все ресурсы процесса. Пример: отдельные пулы потоков или соединений для разных внешних сервисов.

---

**Что такое Strangler pattern?**  
Постепенная миграция монолита в микросервисы без «большого взрыва». Перед монолитом ставится шлюз; новые функции реализуются в микросервисах и подключаются в маршрутизации. По мере переноса трафик переключается на новые сервисы, старый код в монолите отключается. Монолит со временем «обвит» и заменён.

---

**Что такое event-driven архитектура в микросервисах?**  
Взаимодействие через события: сервис публикует событие (что произошло), другие сервисы подписаны и обрабатывают его асинхронно через брокер (Kafka, RabbitMQ). Плюсы: слабая связность, буферизация нагрузки, возможность добавлять потребителей без изменения производителя.

---

**Что такое CQRS?**  
Разделение ответственности: отдельная модель для записи (commands) и для чтения (queries). Write- и read-модели могут использовать разные хранилища и схемы. Позволяет независимо масштабировать и оптимизировать чтение и запись; часто сочетается с event-driven и проекциями из событий.

---

**Что такое Event Sourcing?**  
Хранение не текущего состояния сущности, а последовательности событий (фактов). Состояние получают применением этих событий к пустой модели (replay). Даёт полную историю и аудит, возможность строить новые read-модели из старых событий; усложняет запросы и версионирование событий.
