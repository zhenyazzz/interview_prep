# Архитектура микросервисов — подготовка к собеседованию (Java Backend)

Руководство по микросервисной архитектуре в контексте Java backend и экосистемы Spring.

---

## 1. Введение в микросервисы

### Что такое микросервисная архитектура

**Микросервисная архитектура** — подход к построению приложения как набора **независимых сервисов**, каждый из которых:

- выполняет одну бизнес-область (или ограниченный набор функций);
- развёртывается отдельно;
- имеет собственное хранилище данных (или явно владеет своей частью данных);
- взаимодействует с другими сервисами по сети (HTTP, сообщения).

Сервисы обычно небольшие по коду и по зоне ответственности, но не обязательно «микро» по размеру — важно **независимое развёртывание** и **чёткие границы**.

### Зачем появились микросервисы

- **Масштаб** — большие монолиты сложно разрабатывать, тестировать и развёртывать; одна ошибка может положить всё приложение.
- **Команды** — нужна возможность работать разным командам над разными частями системы без постоянной координации релизов.
- **Технологии** — возможность выбирать стек (язык, БД) под задачу сервиса.
- **Масштабирование** — масштабировать только «узкие» сервисы, а не весь монолит.

### Эволюция: от монолита к микросервисам

```
Монолит (всё в одном процессе)
    → Модульный монолит (логические модули, один деплой)
    → Мультисервис / SOA (сервисы, но часто общая БД и жёсткая связность)
    → Микросервисы (независимые сервисы, своя БД на сервис, слабая связность)
```

Микросервисы — не замена монолита «по умолчанию», а один из вариантов архитектуры при росте системы и команды.

### Цели микросервисной архитектуры

- **Независимое развёртывание** — релиз одного сервиса без перезапуска остальных.
- **Устойчивость** — падение одного сервиса не должно ронять всю систему (изоляция отказов).
- **Масштабируемость** — горизонтальное масштабирование по сервисам.
- **Автономия команд** — команда владеет полным циклом своего сервиса (dev, тесты, деплой).
- **Гибкость технологий** — разные сервисы могут использовать разные языки, БД, фреймворки.

### Краткий ответ для собеседования

**«Что такое микросервисы?»** — Приложение разбито на независимые сервисы. Каждый сервис отвечает за свою доменную область, развёртывается отдельно, имеет свою БД (или чёткое владение данными) и общается с другими по сети. Цель — независимый релиз, изоляция отказов и возможность масштабировать части системы по отдельности.

## 2. Монолитная архитектура и микросервисы

### Монолитная архитектура

Одно приложение — один процесс (или один деплой). Вся бизнес-логика, веб-слой и доступ к БД находятся в одном артефакте. Одна база данных, один релиз, одно масштабирование (как правило, реплики одного и того же приложения).

### Микросервисная архитектура

Приложение состоит из множества сервисов. Каждый сервис — отдельный процесс (или набор реплик), со своей зоной ответственности и, как правило, своей БД. Сервисы взаимодействуют по HTTP/gRPC или через очереди сообщений.

### Сравнительная таблица

| Критерий | Монолит | Микросервисы |
|----------|---------|--------------|
| **Развёртывание** | Один артефакт, один деплой для всего | Деплой по сервисам независимо |
| **Масштабирование** | Масштабируется всё приложение целиком | Масштабируются отдельные сервисы |
| **Скорость разработки** | Проще на старте; со временем замедляется из-за размера кодовой базы | Сложнее на старте; команды могут двигаться быстрее при зрелых процессах |
| **Сложность** | Ниже: один процесс, одна БД, проще отладка | Выше: распределённая система, сеть, согласованность данных |
| **Изоляция отказов** | Сбой в одном модуле может ронять весь процесс | Отказ изолирован в одном сервисе (при правильной настройке) |
| **Организация команд** | Одна большая кодовая база, общий релиз | Команды владеют сервисами, независимые релизы |

### Простая схема (ASCII)

**Монолит:**

```
                    ┌─────────────────────────────────────┐
                    │           Монолит                    │
  Клиент  ────────► │  UI │ Logic │ API │ DB Access │     │
                    │              одна БД ◄───────────────┤
                    └─────────────────────────────────────┘
```

**Микросервисы:**

```
  Клиент
     │
     ▼
┌─────────────┐     HTTP / сообщения
│ API Gateway │ ◄──────────────────────────────────────────►
└──────┬──────┘
       │
   ┌───┴───┬───────────┬───────────┐
   ▼       ▼           ▼           ▼
┌─────┐ ┌─────┐   ┌─────┐   ┌─────┐
│ A   │ │ B   │   │ C   │   │ D   │   ...
│ DB_A│ │ DB_B│   │ DB_C│   │ DB_D│
└─────┘ └─────┘   └─────┘   └─────┘
```

## 3. Характеристики микросервисов

### Небольшие независимые сервисы

- Сервис решает **ограниченную бизнес-задачу** (например, заказы, каталог, уведомления).
- Размер — такой, чтобы одна команда могла понимать и поддерживать сервис; не обязательно «микро» по количеству кода.

### Принцип единственной ответственности (SRP)

- Один сервис — одна зона ответственности в домене.
- Изменения в одной области затрагивают в основном один сервис, что упрощает эволюцию и релизы.

### Независимое развёртывание

- Каждый сервис собирается и выкатывается отдельно.
- Не нужен общий «релиз дня» для всего продукта; уменьшается риск и ускоряется доставка изменений.

### Децентрализованное управление данными

- У сервиса **своя БД** (или свой набор таблиц/схема), к которой не обращаются напрямую другие сервисы.
- Данные других областей получают только через API или события, что снижает связность и даёт возможность менять схему внутри сервиса.

### Изоляция отказов

- При падении одного сервиса остальные продолжают работать (с деградацией функциональности при необходимости).
- Достигается за счёт таймаутов, circuit breaker, fallback и правильной обработки ошибок.

### Масштабируемость

- Можно масштабировать только «узкие» сервисы (больше реплик), не поднимая весь монолит.
- Позволяет экономить ресурсы и лучше реагировать на нагрузку.

## 4. Преимущества микросервисов

| Преимущество | Практический смысл |
|--------------|--------------------|
| **Независимый деплой** | Релиз фичи в одном сервисе без полного регресса и перезапуска всей системы. |
| **Масштабирование** | Добавление реплик только для нагруженных сервисов (например, поиск или расчёт цен). |
| **Автономия команд** | Команда владеет своим сервисом: код, тесты, деплой, мониторинг — меньше блокировок между командами. |
| **Разнообразие технологий** | Разные сервисы могут использовать разные языки, БД, очереди (в разумных пределах). |
| **Изоляция отказов** | Падение сервиса уведомлений не блокирует оформление заказа; можно ввести fallback и ограничение распространения сбоя. |

## 5. Недостатки и сложности микросервисов

### Сложность распределённой системы

- Сеть ненадёжна: задержки, таймауты, частичные сбои.
- Нужны retry, таймауты, идемпотентность, корректная обработка ошибок.
- Отладка и воспроизведение сценариев сложнее, чем в одном процессе.

### Накладные расходы на сеть

- Взаимодействие по HTTP/gRPC или через брокер медленнее, чем вызов метода в памяти.
- Больше сериализации/десериализации, больше нагрузка на сеть и инфраструктуру.

### Сложность отладки в распределённой среде

- Один запрос пользователя может проходить через несколько сервисов. Нужны **распределённая трассировка** и **централизованные логи** с общим идентификатором запроса (trace id, correlation id).

### Операционная сложность

- Больше компонентов: сервисы, очереди, шлюзы, реестры, конфигурация.
- Нужны оркестрация (Kubernetes и т.п.), мониторинг, алерты, стандартизированный деплой и откат.

### Согласованность данных

- Нет единой транзакции между сервисами. Приходится использовать **eventual consistency**, **Saga**, компенсирующие операции, идемпотентность потребителей событий.

### Когда микросервисы — не лучший выбор

- **Небольшая команда и продукт** — монолит проще разрабатывать и эксплуатировать.
- **Жёсткие требования к консистентности** — распределённые транзакции и Saga усложняют модель.
- **Нет зрелых процессов** — CI/CD, мониторинг, трассировка; без них микросервисы превращаются в операционный кошмар.
- **Простой домен** — нет выигрыша от разбиения, зато появляются все издержки распределённой системы.

## 6. Взаимодействие микросервисов

### Синхронное взаимодействие

Клиент (или другой сервис) отправляет запрос и **ждёт ответа**. Типично — HTTP/REST, реже gRPC.

**Плюсы:**

- Простая модель: запрос–ответ, легко отлаживать.
- Понятная семантика (получить данные «здесь и сейчас»).

**Минусы:**

- Связанность по времени: если сервис недоступен или медленный, вызывающий блокируется.
- Каскадные сбои при падении или перегрузке одного сервиса без таймаутов и circuit breaker.

**Примеры:** REST API (Spring MVC, Spring WebFlux), gRPC, OpenFeign между сервисами.

### Асинхронное взаимодействие

Отправитель отправляет сообщение/событие и не ждёт немедленного ответа (или ответ приходит отдельным сообщением). Обычно через **брокер сообщений** или **событийную шину**.

**Плюсы:**

- Слабая связность по времени: производитель и потребитель могут работать независимо.
- Буферизация нагрузки, сглаживание пиков.
- Упрощение интеграции: новые потребители подписываются на события без изменения производителя.

**Минусы:**

- Сложнее отладка и трассировка.
- Нужна обработка дублей (идемпотентность), порядок сообщений (где важно), мониторинг очередей.

**Примеры:** RabbitMQ, Apache Kafka, Redis Streams. В Spring — Spring AMQP, Spring Kafka.

### Сравнение

| Критерий | Синхронное (REST, gRPC) | Асинхронное (Kafka, RabbitMQ) |
|----------|--------------------------|--------------------------------|
| Связность | Вызывающий зависит от доступности сервиса | Слабая связность по времени |
| Консистентность | Проще получить «актуальные» данные | Часто eventual consistency |
| Масштабирование | Нужны таймауты и резервные пути | Очередь как буфер, масштабирование потребителей |
| Сложность | Проще модель запрос–ответ | Сложнее: доставка, повторы, порядок |

На практике часто комбинируют: синхронные вызовы для «запроса данных», асинхронные события для уведомлений и согласования состояния между сервисами.

## 7. Управление данными в микросервисах

### Database per service (своя БД на сервис)

- У каждого сервиса **собственная БД** (или отдельная схема/набор таблиц).
- Другие сервисы **не обращаются к этой БД напрямую** — только через API или события.
- Плюсы: независимость схемы, независимый деплой и масштабирование по данным. Минусы: нет распределённых ACID-транзакций, нужны другие механизмы согласованности.

### Децентрализованное владение данными

- Сервис — единственный владелец своих данных: только он их изменяет и экспортирует через API или события.
- Дублирование данных в других сервисах допустимо через копии (читаемые реплики, кэш, копии по событиям) при явном контракте и понимании eventual consistency.

### Проблема распределённых транзакций

- Классические 2PC/распределённые транзакции в микросервисах обычно не используют: сложность и связность высоки.
- Вместо этого: **eventual consistency**, **Saga**, компенсирующие действия, идемпотентность.

### Eventual consistency ( eventual согласованность)

- Система со временем приходит к согласованному состоянию после распространения событий/обновлений.
- Клиенты и сервисы должны быть готовы к временной несогласованности (повторные запросы, обновление по событиям).

### Saga

- **Saga** — последовательность локальных транзакций в разных сервисах. При сбое выполняются **компенсирующие транзакции** (откат в обратном порядке или компенсирующие действия).
- Хореография: сервисы реагируют на события и сами запускают следующие шаги и компенсации. Оркестрация: отдельный оркестратор вызывает сервисы и решает, что делать при ошибке.

### API Composition

- Когда нужны данные из нескольких сервисов (например, заказ + товары + пользователь), **API Composition** — вызов нескольких сервисов из одного (или из API Gateway/BFF) и объединение результатов. Альтернатива — заранее подготовленные денормализованные данные через события (CQRS-подходы, read-модели).

## 8. Service Discovery (обнаружение сервисов)

### Зачем нужен Service Discovery

В облаке сервисы масштабируются, меняют IP и порты, поднимаются и останавливаются. Жёстко прописывать адреса в конфигурации нельзя. **Service Discovery** даёт возможность находить экземпляры сервиса по имени (логическому) и получать их сетевые адреса.

### Service Registry (реестр сервисов)

- Центральный (или распределённый) реестр, в котором хранятся **имя сервиса → список инстансов (host:port)**.
- Сервисы при старте **регистрируются** в реестре и периодически обновляют свой статус (heartbeat). При остановке — снимаются с регистрации (или TTL).

### Регистрация и поиск

- **Регистрация:** при старте сервис отправляет в реестр свой адрес и метаданные (версия, зона и т.д.).
- **Поиск:** клиент (или прокси/шлюз) запрашивает у реестра список инстансов по имени сервиса и выбирает один (round-robin, random, по зоне и т.д.) — это **client-side discovery**. Либо запрос идёт через балансировщик/прокси, который сам обращается к реестру — **server-side discovery**.

### Примеры

| Решение | Описание |
|---------|----------|
| **Eureka** (Netflix / Spring Cloud) | Реестр с REST API. Клиенты регистрируются и получают список инстансов; часто используется с Ribbon/LoadBalancer для выбора инстанса. |
| **Consul** (HashiCorp) | Реестр + проверки здоровья, KV, multi-datacenter. Поддерживается Spring Cloud Consul. |
| **Kubernetes** | Встроенный discovery: у каждого Service есть DNS-имя (например, `http://order-service`). Поды регистрируются автоматически через kubelet. Отдельный реестр не обязателен. |

В Spring Cloud типичный вариант: сервисы регистрируются в Eureka; вызовы между сервисами идут по имени (например, через RestTemplate/WebClient с LoadBalancer или OpenFeign с тем же механизмом).

## 9. API Gateway

### Роль API Gateway

**API Gateway** — единая точка входа для клиентов (браузер, мобильное приложение, партнёры). Клиенты обращаются только к шлюзу; шлюз маршрутизирует запросы к нужным микросервисам, агрегирует ответы при необходимости и применяет сквозные функции (аутентификация, лимиты, логирование).

### Основные функции

| Функция | Описание |
|---------|----------|
| **Единая точка входа** | Один URL/домен для клиента; внутренняя топология сервисов скрыта. |
| **Маршрутизация** | По пути, заголовкам, методу — перенаправление на соответствующий сервис. |
| **Аутентификация и авторизация** | Проверка JWT/OAuth2, проверка прав доступа до попадания в сервисы. |
| **Ограничение частоты запросов (rate limiting)** | Защита от перегрузки и злоупотреблений. |
| **Агрегация ответов** | Один запрос клиента → несколько вызовов к сервисам → объединённый ответ (BFF-подход). |
| **Терминация SSL, сжатие** | Централизованная обработка на границе. |

### Примеры

| Решение | Особенности |
|---------|-------------|
| **Spring Cloud Gateway** | Маршрутизация на основе предикатов и фильтров, интеграция с Spring Security, реактивный стек. Удобен в экосистеме Spring. |
| **NGINX** | Reverse proxy, балансировка, кэширование, часто используется как входной прокси перед шлюзом или вместо простого шлюза. |
| **Kong** | API Gateway на OpenResty (NGINX + Lua), плагины для аутентификации, rate limiting, логирования. |

## 10. Управление конфигурацией

### Централизованная конфигурация

В микросервисах много инстансов; менять конфиг на каждой машине неудобно и ошибкоопасно. **Централизованная конфигурация** — хранилище настроек (файлы, БД, vault), откуда сервисы при старте (или по обновлению) получают конфигурацию по своему имени и профилю (например, `order-service`, профиль `prod`).

### Spring Cloud Config

- **Config Server** — отдельный сервис, отдающий конфигурацию из Git, файловой системы или Vault.
- Клиенты (микросервисы) при старте обращаются к Config Server по идентификатору приложения и профилю и получают `application.yml` / properties.
- Поддержка профилей (dev, prod), шифрование чувствительных значений, обновление без перезапуска (например, через Spring Cloud Bus и `/actuator/refresh`).

### Конфигурация по окружениям

- Один и тот же образ приложения может использоваться в dev/stage/prod; различие — в профиле и данных, полученных из Config Server или переменных окружения (Twelve-Factor: конфиг в окружении/вне образа).

## 11. Устойчивость к сбоям (Fault Tolerance)

В распределённой системе отказы неизбежны. Цель — не допустить каскадных падений и дать пользователю приемлемый ответ (или понятную ошибку).

| Приём | Описание |
|-------|----------|
| **Retry** | Повтор запроса при временной ошибке (с ограничением числа попыток и backoff). |
| **Timeout** | Ограничение времени ожидания ответа, чтобы не блокировать поток и не накапливать зависшие вызовы. |
| **Circuit Breaker** | При множестве ошибок «размыкает цепь» и временно не вызывает сбойный сервис (сразу возвращает ошибку или fallback), давая ему восстановиться. Периодически проверяет доступность (half-open). |
| **Fallback** | При ошибке или открытом circuit breaker вернуть запасное значение, кэш или упрощённый ответ вместо полного отказа. |

### Resilience4j

Библиотека для Java (в т.ч. Spring Boot): circuit breaker, retry, rate limiter, bulkhead, таймауты. Используется вместо устаревшего Hystrix (снят с разработки). Интеграция с Spring Cloud и обычным Spring Boot через аннотации и конфигурацию.

### Hystrix (историческая справка)

Netflix Hystrix раньше использовался для circuit breaker и fallback в Spring Cloud. Сейчас в режиме поддержки; в новых проектах рекомендуется Resilience4j или возможности платформы (Kubernetes, mesh).

## 12. Логирование и мониторинг в микросервисах

### Сложность мониторинга распределённой системы

Один запрос пользователя проходит через несколько сервисов. Нужно:

- понимать, где произошла ошибка или задержка;
- видеть полный путь запроса (трассировка);
- агрегировать логи и метрики из всех сервисов.

### Централизованное логирование

- Логи со всех инстансов собираются в одно хранилище (например, ELK: Elasticsearch, Logstash, Filebeat).
- В логах обязательно **correlation/trace id**, чтобы собрать все записи одного запроса.
- Поиск и анализ через Kibana или аналог.

### Распределённая трассировка (Distributed Tracing)

- Каждый запрос получает **trace id** (и span id для шагов). При переходе между сервисами id передаётся в заголовках.
- Трассировочная система (Zipkin, Jaeger, Sleuth/Micrometer) собирает span’ы и строит дерево вызовов: какой сервис сколько времени занял, где была ошибка.

### Метрики

- Сбор метрик с каждого сервиса (throughput, latency, ошибки, использование ресурсов). Типичный стек: **Prometheus** (сбор и хранение) + **Grafana** (дашборды и алерты). В Spring — Micrometer, который отдаёт метрики в Prometheus или другую систему.

### Инструменты (кратко)

| Инструмент | Назначение |
|------------|------------|
| **ELK (Elasticsearch, Logstash, Kibana)** | Хранение и поиск логов, визуализация. |
| **Prometheus** | Сбор метрик, запросы (PromQL), алертинг. |
| **Grafana** | Дашборды по метрикам и логам, алерты. |
| **Zipkin / Jaeger** | Распределённая трассировка (сбор и отображение trace). |
| **Spring Cloud Sleuth / Micrometer Tracing** | Встраивание trace id в логи и отправка span’ов в Zipkin/Jaeger. |

## 13. Безопасность в микросервисах

### Аутентификация и авторизация

- **Аутентификация** — проверка личности (логин/пароль, токен). В микросервисах часто выносится на границу: API Gateway или отдельный сервис (Identity Provider), который выдаёт токен.
- **Авторизация** — проверка прав на действие/ресурс. Может выполняться в шлюзе (по scope/ролям в токене) или внутри сервиса (проверка прав к конкретному ресурсу).

### Токены (JWT, OAuth2)

- **JWT** — подписанный токен с claims (роли, scope, userId). Сервисы проверяют подпись и не обращаются к центральному серверу при каждом запросе (stateless).
- **OAuth2** — протокол выдачи токенов (authorization code, client credentials и т.д.). Микросервисы часто выступают как Resource Server: проверяют access token, выданный Authorization Server.

### Безопасность на API Gateway

- Терминация TLS, проверка JWT/OAuth2, проверка scope, rate limiting, блокировка нежелательных запросов. Внутренняя сеть между шлюзом и сервисами может быть закрыта от внешнего мира; сервисы могут доверять заголовкам, установленным шлюзом (при правильной настройке сети).

## 14. Микросервисы на Spring Boot и Spring Cloud

Типичный стек в Java для микросервисов:

| Компонент | Технология | Назначение |
|-----------|------------|------------|
| Приложение | **Spring Boot** | Быстрый старт сервиса, встроенный сервер, конфигурация, метрики. |
| Реестр | **Eureka** (Spring Cloud Netflix) | Регистрация и обнаружение сервисов. |
| Вызовы между сервисами | **OpenFeign** / **RestTemplate** + LoadBalancer | HTTP-клиент с подстановкой адреса из Eureka по имени сервиса. |
| Шлюз | **Spring Cloud Gateway** | Маршрутизация, фильтры, интеграция с Spring Security. |
| Конфигурация | **Spring Cloud Config** | Централизованный конфиг из Git/файлов. |
| Устойчивость | **Resilience4j** | Circuit breaker, retry, таймауты. |
| Сообщения | **Spring Kafka** / **Spring AMQP** | Асинхронное взаимодействие через Kafka или RabbitMQ. |
| Трассировка | **Micrometer Tracing**, **Zipkin** | Распределённая трассировка и интеграция с логами. |

### Упрощённая схема

```
                    ┌──────────────────┐
                    │  Spring Cloud    │
  Клиент ──────────►│  Gateway         │──────────────► Eureka (Discovery)
                    └────────┬─────────┘
                             │
         ┌───────────────────┼───────────────────┐
         ▼                   ▼                    ▼
  ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
  │ Order       │    │ Product     │    │ User        │
  │ Service     │◄──►│ Service     │    │ Service     │
  │ (Feign →…)  │    │             │    │             │
  └──────┬──────┘    └──────┬──────┘    └──────┬──────┘
         │                  │                  │
         ▼                  ▼                  ▼
      [БД заказов]     [БД товаров]      [БД пользователей]
```

Сервисы регистрируются в Eureka; Gateway и другие сервисы вызывают их по имени. Внутри сервисов — Spring Boot, REST, БД, при необходимости Kafka/RabbitMQ для событий.

## 15. Когда использовать микросервисы

### Микросервисы уместны при:

- **Крупной системе и нескольких командах** — нужны независимые релизы и владение частями продукта.
- **Сложном домене** — разбиение по bounded context (DDD) естественно ложится на сервисы.
- **Высоких требованиях к масштабированию** — разные части системы нагружены по-разному, нужно масштабировать выборочно.
- **Разных технологических требованиях** — разные БД, языки или стеки для разных подсистем (в разумных границах).

### Монолит может быть лучше при:

- **Небольшой команде и продукте** — меньше операционной нагрузки, проще разработка и отладка.
- **Жёстких требованиях к консистентности** — проще одна БД и ACID-транзакции.
- **Ранней стадии продукта** — быстрее итерировать в монолите; разбивать на сервисы можно при появлении чётких границ и потребности в независимом деплое.
- **Отсутствии зрелой DevOps/мониторинга** — без автоматизации и наблюдаемости микросервисы сильно усложняют жизнь.

## 16. Краткое резюме для собеседования

### Микросервисы

- Набор **независимых сервисов**, каждый со своей зоной ответственности и своей БД (или владением данными).
- Взаимодействие по **сети** (HTTP/REST, gRPC или очереди/события).
- Цели: **независимый деплой**, изоляция отказов, выборочное масштабирование, автономия команд.

### Монолит vs микросервисы

- Монолит: один процесс, один деплой, проще на старте; сложнее масштабировать и эволюционировать при росте.
- Микросервисы: независимый деплой и масштабирование, но выше сложность (распределённость, данные, операции).

### Ключевые компоненты

- **Service Discovery** (Eureka, Consul, K8s) — поиск инстансов по имени.
- **API Gateway** — единая точка входа, маршрутизация, аутентификация, rate limiting.
- **Config Server** — централизованная конфигурация (например, Spring Cloud Config).
- **Устойчивость** — retry, timeout, circuit breaker, fallback (Resilience4j).
- **Наблюдаемость** — централизованные логи (ELK), метрики (Prometheus, Grafana), трассировка (Zipkin, Jaeger).

### Данные

- **Database per service**, eventual consistency, **Saga** для распределённых сценариев, **API Composition** или подготовленные read-модели для данных из нескольких сервисов.

### Связь

- **Синхронная** (REST, gRPC) — запрос–ответ; проще, но связность по доступности.
- **Асинхронная** (Kafka, RabbitMQ) — события/очереди; слабая связность по времени, сложнее отладка и согласованность.

### Spring-экосистема

- **Spring Boot** — основа сервиса; **Spring Cloud** — Eureka, Gateway, Config, OpenFeign, интеграция с Resilience4j и трассировкой. Типичный выбор для Java-микросервисов.
