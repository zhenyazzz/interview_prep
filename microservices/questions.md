# Вопросы и ответы: микросервисы (собеседование Java Backend)

Часто задаваемые вопросы по микросервисной архитектуре с краткими ответами для подготовки к техническим собеседованиям.

---

# Основные вопросы о микросервисах

## Что такое микросервисы?

Микросервисы — архитектурный подход, при котором приложение строится как набор **независимых сервисов**. Каждый сервис отвечает за свою бизнес-область, развёртывается отдельно, имеет свою БД (или владеет своей частью данных) и общается с другими по сети (HTTP или сообщения). Цель — независимый релиз, изоляция отказов и выборочное масштабирование.

## Чем микросервисы отличаются от монолита?

- **Монолит** — одно приложение, один процесс, один деплой, одна БД. Всё масштабируется и падает вместе.
- **Микросервисы** — много сервисов, каждый со своим деплоем и своей БД. Сервисы связаны сетью; можно масштабировать и менять части системы по отдельности.

## Почему компании переходят на микросервисную архитектуру?

- Необходимость **независимых релизов** разными командами без блокировок.
- **Масштабирование** только нагруженных частей системы, а не всего приложения.
- **Изоляция отказов** — падение одного сервиса не роняет всю систему.
- **Автономия команд** — команда владеет полным циклом своего сервиса (разработка, тесты, деплой).

## Какие основные характеристики микросервисов?

- Небольшие **независимые сервисы** с одной зоной ответственности (SRP).
- **Независимый деплой** — релиз одного сервиса без перезапуска остальных.
- **Своя БД на сервис** (или чёткое владение данными).
- Взаимодействие по **сети** (синхронно — REST/gRPC, асинхронно — очереди/события).
- **Изоляция отказов** и возможность **масштабировать по сервисам**.

## Какие проблемы решают микросервисы?

- Долгие и рискованные релизы монолита — заменяются частыми независимыми релизами.
- Невозможность масштабировать только «узкое место» — решается выборочным масштабированием сервисов.
- Блокировки между командами при общей кодовой базе — снижаются за счёт владения сервисами.
- Каскадные падения при сбое одного модуля — ограничиваются изоляцией сервисов и устойчивостью (circuit breaker, timeout).

# Monolith vs Microservices

## Что такое монолитная архитектура?

Монолит — приложение, в котором вся бизнес-логика, веб-слой и доступ к данным находятся в **одном процессе** (одном артефакте). Одна база данных, один деплой, одно масштабирование (реплики одного приложения).

## Какие преимущества у монолита?

- **Проще разработка и отладка** — один процесс, одна БД, полные транзакции.
- **Проще деплой** — один артефакт, меньше компонентов.
- **Меньше операционной сложности** — не нужны реестры, шлюзы, распределённая трассировка на старте.
- **Подходит для небольших команд и продуктов** на ранней стадии.

## Какие проблемы возникают в монолитах?

- **Один деплой на всё** — любое изменение требует полного регресса и координации.
- **Масштабирование только целиком** — нельзя масштабировать только «узкое» место.
- **Растущая связность кода** — сложнее поддерживать и менять при росте кодовой базы.
- **Единая точка отказа** — сбой в одном модуле может ронять весь процесс.

## Когда лучше использовать монолит?

- Небольшая команда и продукт на стадии поиска продукта (MVP).
- Простой домен без чётких границ для разбиения на сервисы.
- Жёсткие требования к консистентности данных (удобнее одна БД и ACID).
- Нет зрелых процессов CI/CD и мониторинга — микросервисы без этого усложняют жизнь.

## Таблица сравнения: Монолит vs Микросервисы

| Критерий | Монолит | Микросервисы |
|----------|---------|--------------|
| **Деплой** | Один артефакт, один релиз для всего | Независимый деплой по сервисам |
| **Масштабирование** | Всё приложение целиком | Выборочно по сервисам |
| **Сложность** | Ниже: один процесс, одна БД | Выше: сеть, данные, операции |
| **Отказоустойчивость** | Сбой в модуле может ронять весь процесс | Изоляция: падает один сервис |
| **Разработка** | Проще на старте; при росте — медленнее | Сложнее на старте; команды могут двигаться быстрее при зрелых процессах |

# Коммуникация между микросервисами

## Как микросервисы общаются между собой?

Двумя основными способами: **синхронно** (запрос–ответ по HTTP/REST или gRPC) и **асинхронно** (через брокер сообщений — Kafka, RabbitMQ). На практике часто комбинируют: синхронные вызовы для «получить данные сейчас», асинхронные события для уведомлений и согласования состояния между сервисами.

## Что такое синхронная коммуникация?

Клиент (сервис) отправляет запрос и **ждёт ответа**. Типично — REST API или gRPC. Плюсы: простая модель, понятная семантика. Минусы: связность по доступности — если сервис недоступен или медленный, вызывающий блокируется; при массовых сбоях возможны каскадные падения без таймаутов и circuit breaker.

## Что такое асинхронная коммуникация?

Отправитель публикует сообщение/событие и **не ждёт немедленного ответа**. Получатели обрабатывают сообщения из очереди или топика. Плюсы: слабая связность по времени, буферизация нагрузки. Минусы: сложнее отладка и трассировка, нужна обработка дублей (идемпотентность) и понимание eventual consistency.

## Когда использовать REST?

- Когда нужен **запрос–ответ** и актуальные данные «здесь и сейчас» (получить заказ, проверить наличие).
- Простая интеграция, широко поддерживаемые инструменты и клиенты.
- Подходит для вызовов между сервисами при наличии таймаутов, retry и circuit breaker.

## Когда использовать Kafka или message broker?

- **События** и уведомления (заказ создан, платёж проведён) — потребители подписываются и обрабатывают асинхронно.
- **Слабая связность** — производитель не знает потребителей; добавление новых не меняет производителя.
- **Высокая нагрузка и буферизация** — очередь сглаживает пики и позволяет масштабировать потребителей.
- **Event sourcing, CQRS** — поток событий как источник правды.

# Service Discovery

## Что такое Service Discovery?

**Service Discovery** — механизм, позволяющий находить **инстансы сервисов по имени** (логическому), а не по фиксированному IP. В облаке и Kubernetes инстансы создаются и удаляются, меняются адреса; реестр хранит актуальный список инстансов, куда можно направить запрос.

## Почему service discovery необходим в микросервисах?

Инстансы сервисов **динамически** меняют IP и порты (масштабирование, перезапуски, оркестрация). Жёстко прописанные адреса в конфигурации делают систему хрупкой и не позволяют масштабировать и восстанавливаться без ручной смены конфигов. Discovery даёт обращение по имени сервиса и автоматическое обновление списка инстансов.

## Что такое service registry?

**Service registry** (реестр сервисов) — хранилище «имя сервиса → список инстансов (host:port)». Сервисы при старте регистрируются в реестре и периодически обновляют статус (heartbeat). Клиенты или шлюз запрашивают у реестра список инстансов по имени и выбирают один для вызова.

## Что такое service registration?

**Service registration** — процесс, при котором сервис при **старте** сообщает реестру свой адрес и метаданные (версия, зона). Периодически отправляет heartbeat; при остановке снимается с регистрации или запись истекает по TTL. Без регистрации реестр не знает, куда направлять трафик.

## Чем отличается client-side discovery от server-side discovery?

- **Client-side:** клиент (или SDK в сервисе) **сам** обращается к реестру, получает список инстансов и выбирает один (round-robin, random). Пример: Eureka + LoadBalancer в Spring.
- **Server-side:** клиент обращается к **фиксированному адресу** (балансировщику/прокси); тот обращается к реестру и направляет запрос на инстанс. Пример: Kubernetes Service — клиент зовёт по имени сервиса, kube-proxy маршрутизирует на поды.

## Примеры: Eureka, Consul, Kubernetes

| Решение | Описание |
|---------|----------|
| **Eureka** | Реестр Netflix; Spring Cloud интегрирует регистрацию при старте и клиентский выбор инстанса (RestTemplate/OpenFeign с LoadBalancer). |
| **Consul** | Реестр HashiCorp с health checks, KV, multi-datacenter; Spring Cloud Consul. |
| **Kubernetes** | Встроенный discovery: у каждого Service есть DNS-имя (например, `http://order-service`); поды автоматически участвуют в сервисе, отдельный реестр не обязателен. |

# API Gateway

## Что такое API Gateway?

**API Gateway** — единая точка входа для клиентов (веб, мобильные приложения, партнёры). Все запросы идут на шлюз; шлюз маршрутизирует их к нужным микросервисам и выполняет сквозные функции: аутентификация, лимиты, агрегация ответов. Внутренняя топология сервисов скрыта от клиента.

## Какие задачи выполняет API Gateway?

- **Маршрутизация** — по URL, заголовкам, методу направлять запрос на нужный сервис.
- **Аутентификация и авторизация** — проверка JWT/OAuth2, проверка прав доступа.
- **Rate limiting** — ограничение частоты запросов по клиенту/ключу.
- **Агрегация** — один запрос клиента → несколько вызовов к сервисам → объединённый ответ (BFF).
- Терминация SSL, сжатие, логирование, кэширование.

## Почему API Gateway используется в микросервисах?

Клиенту не нужно знать десятки URL и сервисов; один домен/точка входа. Аутентификация, лимиты и логирование делаются в одном месте, а не в каждом сервисе. Упрощается версионирование API и смена внутренней структуры без смены контракта для клиента.

## Какие проблемы он решает?

- Скрытие внутренней структуры и количества сервисов.
- Централизованная аутентификация и авторизация на границе.
- Защита от перегрузки (rate limiting) и злоупотреблений.
- Упрощение клиентского кода — один базовый URL, при необходимости агрегированные ответы.

## Примеры технологий

| Технология | Особенности |
|------------|-------------|
| **Spring Cloud Gateway** | Маршруты по предикатам и фильтрам, интеграция с Spring Security, реактивный стек; удобен в экосистеме Spring. |
| **Kong** | Шлюз на OpenResty (NGINX + Lua), плагины для аутентификации, rate limiting, логирования. |
| **NGINX** | Reverse proxy, балансировка, кэш; часто как входной прокси перед приложением или как простой шлюз. |

# Управление данными

## Почему в микросервисах используется принцип database per service?

Чтобы каждый сервис **независимо** владел своими данными: своя схема, свои миграции, свой деплой. Прямой доступ других сервисов к этой БД создаёт жёсткую связность и блокирует независимые изменения. Доступ к данным других областей — только через API или события.

## Почему нельзя использовать одну общую базу данных?

- **Связность** — изменение схемы затрагивает несколько команд и сервисов; сложно эволюционировать.
- **Единая точка отказа** — падение БД роняет всё приложение.
- **Невозможность независимого деплоя и масштабирования** по сервисам.
- Нарушается граница владения данными и доменных контекстов (DDD).

## Какие проблемы возникают с консистентностью данных?

Нет единой ACID-транзакции между сервисами. Изменения в нескольких сервисах согласуются через **события** или **компенсирующие действия** (Saga), что приводит к **eventual consistency**: система приходит к согласованному состоянию не мгновенно, а со временем. Нужна идемпотентность потребителей и обработка дублей.

## Eventual consistency

**Eventual consistency** — модель, при которой после распространения обновлений (через события или репликацию) система **в итоге** приходит к согласованному состоянию. В момент времени возможна временная несогласованность (например, прочитанные данные чуть устарели). Клиенты и сервисы должны это учитывать; при необходимости читать повторно или использовать версии/метки времени.

# Распределённые транзакции

## Что такое распределённая транзакция?

Операция, которая затрагивает **несколько сервисов (и их БД)** и с точки зрения бизнеса должна быть выполнена «целиком или откачена целиком». В монолите это одна транзакция; в микросервисах у каждого сервиса своя БД — единой транзакции нет, нужны другие механизмы (Saga, компенсации).

## Почему 2PC редко используется в микросервисах?

**2PC (two-phase commit)** требует координатора и блокировок участников. При падениях узлов и сети это даёт длительные блокировки и сложное восстановление. Микросервисы ориентированы на слабую связность и отказоустойчивость; 2PC создаёт общую точку отказа и усложняет архитектуру. Поэтому предпочитают Saga и eventual consistency.

## Что такое Saga pattern?

**Saga** — последовательность **локальных транзакций** в разных сервисах. При сбое на любом шаге выполняются **компенсирующие транзакции** (откат в бизнес-смысле), а не откат БД другого сервиса. Два подхода: **хореография** (сервисы реагируют на события и сами запускают следующие шаги и компенсации) и **оркестрация** (центральный оркестратор вызывает сервисы и при ошибке запускает компенсации).

## Какие существуют типы Saga?

- **Choreography (хореография)** — нет центрального оркестратора; каждый сервис выполняет свой шаг и публикует событие; следующие сервисы подписаны и выполняют свои шаги или компенсации. Слабая связность, сложнее отслеживать поток.
- **Orchestration (оркестрация)** — отдельный оркестратор вызывает сервисы по шагам и при ошибке запускает компенсации в обратном порядке. Явный поток, проще отладка; оркестратор — критичный компонент.

# Отказоустойчивость

## Что такое Circuit Breaker?

**Circuit Breaker** — паттерн, ограничивающий вызовы к сбойному сервису. После серии ошибок «размыкает цепь»: какое-то время **не вызывает** сервис (сразу возвращает ошибку или fallback), давая ему восстановиться. Периодически переходит в half-open и пробует вызовы снова; при успехе — снова разрешает вызовы (closed). Защищает от каскадных сбоев и исчерпания ресурсов у вызывающей стороны. В Java часто используют **Resilience4j**.

## Зачем нужен Retry?

Многие сбои **временные** (таймаут сети, кратковременная перегрузка). **Retry** — автоматический повтор запроса с ограничением числа попыток и задержкой (желательно exponential backoff), чтобы не усугублять перегрузку. Важно: не повторять неидемпотентные операции без идемпотентного ключа на стороне сервиса.

## Что такое Timeout?

**Timeout** — максимальное время ожидания ответа на вызов. По истечении вызов прерывается, ресурсы (поток, соединение) освобождаются. Без таймаутов «зависший» сервис может накапливать ожидающие вызовы и ронять вызывающий сервис. Критично задавать таймауты для всех внешних вызовов (HTTP, БД, очереди).

## Что такое Bulkhead pattern?

**Bulkhead** — изоляция ресурсов (потоки, соединения) по разным зависимостям. Цель: сбой или перегрузка одного сервиса не забирают **все** ресурсы процесса. Примеры: отдельные пулы потоков или соединений для вызовов к разным сервисам; переполнение одного пула не блокирует вызовы к здоровым сервисам. В Java — Resilience4j Bulkhead или отдельные пулы в клиентах.

# Мониторинг и логирование

## Почему логирование в микросервисах сложнее?

Один запрос пользователя проходит через **несколько сервисов**. Логи разбросаны по разным процессам и машинам; без общего идентификатора (trace id, correlation id) невозможно собрать полную картину запроса. Нужны централизованный сбор логов и распределённая трассировка с передачей trace id между сервисами.

## Что такое distributed tracing?

**Distributed tracing** — механизм отслеживания **пути запроса** через несколько сервисов. Каждому запросу присваивается **trace id**; при переходе между сервисами он передаётся в заголовках. Система (Zipkin, Jaeger) собирает **span'ы** (шаги вызовов) и строит дерево: какой сервис сколько времени занял, где произошла ошибка. Упрощает отладку и поиск узких мест.

## Какие инструменты используются для мониторинга микросервисов?

| Категория | Инструменты |
|-----------|--------------|
| Логи | **ELK** (Elasticsearch, Logstash, Kibana), централизованный сбор с trace id |
| Метрики | **Prometheus** (сбор и хранение), **Grafana** (дашборды и алерты) |
| Трассировка | **Zipkin**, **Jaeger**; в Spring — Micrometer Tracing, Sleuth |

Дополнительно: health checks (Actuator), алертинг (Alertmanager, PagerDuty), метрики приложения через **Micrometer**.

## Примеры: ELK, Prometheus, Grafana, Zipkin, Jaeger

| Инструмент | Назначение |
|------------|------------|
| **ELK stack** | Elasticsearch — хранение логов; Logstash/Filebeat — сбор; Kibana — поиск и визуализация. |
| **Prometheus** | Сбор метрик (pull-модель), хранение временных рядов, запросы PromQL, алертинг. |
| **Grafana** | Дашборды по метрикам и логам, алерты; часто используется с Prometheus и Loki. |
| **Zipkin** | Распределённая трассировка: сбор span'ов, отображение дерева вызовов. |
| **Jaeger** | Аналог Zipkin, распределённая трассировка; совместим с OpenTracing/OpenTelemetry. |

# Безопасность микросервисов

## Как реализуется аутентификация в микросервисах?

Часто **на границе** — в API Gateway или отдельном Identity Provider. Пользователь логинится один раз; выдаётся **токен** (JWT или opaque). Дальнейшие запросы содержат токен в заголовке; шлюз (или сервисы) проверяют подпись и claims. Сервисы могут не обращаться к центральному серверу при каждом запросе (stateless при JWT).

## Что такое JWT?

**JWT (JSON Web Token)** — компактный токен из трёх частей (header, payload, signature). Payload содержит **claims** (например, userId, roles, scope). Подписывается секретом или ключом; сервисы проверяют подпись и читают claims без обращения к серверу выдачи. Используется как access token для доступа к API.

## Где лучше проверять токены — в API Gateway или сервисах?

**В API Gateway** — один раз на границе; внутренние сервисы получают уже проверенный запрос (и при необходимости обогащённые заголовки: user id, роли). Плюсы: не дублировать проверку в каждом сервисе, внутренняя сеть может быть закрыта. **В сервисах** — если сервисы могут вызываться минуя шлюз (внутренние вызовы) или нужна проверка прав к конкретному ресурсу; тогда сервис сам проверяет токен или доверяет заголовкам от доверенного шлюза.

## Что такое OAuth2?

**OAuth2** — протокол **авторизации** (выдача токенов доступа к ресурсам). Роли: Resource Owner (пользователь), Client (приложение), Authorization Server (выдаёт токены), Resource Server (микросервисы, проверяют токен). Типичные потоки: Authorization Code (веб), Client Credentials (сервис-сервис). Микросервисы часто выступают как Resource Server: проверяют access token, выданный Authorization Server.

# Микросервисы в экосистеме Spring

## Какие технологии Spring используются для микросервисов?

- **Spring Boot** — основа сервиса (REST, конфигурация, метрики, health).
- **Spring Cloud** — реестр (Eureka), шлюз (Gateway), конфиг (Config), вызовы по имени (OpenFeign, LoadBalancer), устойчивость (Resilience4j), трассировка (Micrometer).
- **Spring Kafka / Spring AMQP** — асинхронное взаимодействие.
- **Spring Security** — аутентификация, OAuth2 Resource Server, JWT.

## Что такое Spring Cloud?

**Spring Cloud** — набор библиотек и решений для типичных задач распределённых систем: обнаружение сервисов (Eureka, Consul), конфигурация (Config), маршрутизация (Gateway), клиенты с балансировкой (LoadBalancer, OpenFeign), устойчивость, трассировка. Интегрируется со Spring Boot и позволяет быстро собрать микросервисный стек на Java.

## Что такое Eureka?

**Eureka** — сервер **service discovery** (Netflix). Сервисы при старте регистрируются в Eureka и периодически обновляют heartbeat. Клиенты запрашивают у Eureka список инстансов по имени сервиса; Spring Cloud LoadBalancer выбирает инстанс и выполняет вызов. Позволяет вызывать сервисы по имени (например, `http://order-service`) без жёстких адресов.

## Что такое Spring Cloud Gateway?

**Spring Cloud Gateway** — реактивный **API Gateway** на Spring WebFlux. Маршрутизация на основе предикатов (path, host, header и т.д.) и фильтров (аутентификация, логирование, rate limiting). Интегрируется с Eureka для маршрутизации по имени сервиса и с Spring Security. Альтернатива Zuul; подходит для современного стека Spring.

## Что такое OpenFeign?

**OpenFeign** — декларативный **HTTP-клиент**: интерфейс с аннотациями методов и путей; Feign генерирует реализацию. В Spring Cloud интегрируется с LoadBalancer и Eureka: вызов по имени сервиса (например, `@FeignClient("order-service")`), балансировка по инстансам, при необходимости retry и circuit breaker. Упрощает вызовы между микросервисами.

# Самые частые вопросы на собеседованиях

## Когда стоит использовать микросервисы?

- Крупная система и **несколько команд**, нужны независимые релизы.
- **Сложный домен** с чёткими границами (bounded context в DDD).
- Высокие требования к **масштабированию** разных частей по-разному.
- Зрелые процессы **CI/CD и мониторинга**; готовность к операционной сложности распределённой системы.

## Когда лучше выбрать монолит?

- **Небольшая команда** и продукт на ранней стадии (MVP).
- **Простой домен** без очевидного разбиения на сервисы.
- Жёсткие требования к **консистентности** (удобнее одна БД и ACID).
- Нет готовности к **DevOps** и наблюдаемости — микросервисы без этого усложняют эксплуатацию.

## Какие основные проблемы микросервисов?

- **Сложность распределённой системы** — сеть, таймауты, частичные сбои.
- **Согласованность данных** — нет единой транзакции; Saga, eventual consistency, идемпотентность.
- **Операционная сложность** — больше компонентов, нужны discovery, шлюз, конфиг, трассировка, мониторинг.
- **Отладка и наблюдение** — один запрос через много сервисов; нужны централизованные логи и distributed tracing.

## Какие паттерны используются в микросервисах?

- **API Gateway** — единая точка входа, маршрутизация, аутентификация.
- **Service Discovery** — динамическое обнаружение инстансов (Eureka, Consul, K8s).
- **Database per Service** — своя БД на сервис.
- **Saga** — распределённые бизнес-транзакции с компенсациями.
- **Circuit Breaker, Retry, Timeout, Bulkhead** — отказоустойчивость.
- **Event-Driven** — асинхронное взаимодействие через события (Kafka, RabbitMQ).
- **CQRS, Event Sourcing** — при необходимости (сложные запросы, полная история).

## Какие проблемы возникают при масштабировании микросервисов?

- **Обнаружение и балансировка** — нужен реестр и актуальные списки инстансов.
- **Конфигурация** — централизованное управление настройками для многих инстансов.
- **Мониторинг и трассировка** — агрегация метрик и логов с большого числа инстансов.
- **Согласованность и дубли** — при репликации и кэшировании; идемпотентность и версионирование данных.

# Быстрый повтор перед собеседованием

## Ключевые концепции микросервисов

- **Независимые сервисы** — свой деплой, своя БД (или владение данными), одна зона ответственности.
- **Взаимодействие по сети** — синхронно (REST, gRPC) или асинхронно (Kafka, RabbitMQ).
- **Цели** — независимый релиз, изоляция отказов, выборочное масштабирование, автономия команд.
- **Database per service** — нет общей БД; согласованность через API и события (eventual consistency).

## Важные паттерны

| Паттерн | Суть |
|---------|------|
| API Gateway | Единая точка входа, маршрутизация, аутентификация, rate limiting |
| Service Discovery | Реестр инстансов по имени; Eureka, Consul, K8s |
| Saga | Распределённые шаги + компенсации при сбое; хореография или оркестрация |
| Circuit Breaker | Ограничение вызовов к сбойному сервису; защита от каскадных сбоев |
| Retry / Timeout | Повторы при временных сбоях; ограничение времени ожидания |
| Bulkhead | Изоляция ресурсов (пулы) по зависимостям |
| Event-Driven | События через брокер; слабая связность, Kafka/RabbitMQ |

## Основные проблемы распределённых систем

- **Сеть** — задержки, таймауты, частичные сбои; нужны retry, timeout, circuit breaker.
- **Данные** — нет распределённых ACID-транзакций; Saga, eventual consistency, идемпотентность.
- **Наблюдаемость** — логи и метрики с многих узлов; нужны централизация и distributed tracing (trace id).
- **Операции** — больше компонентов; нужны автоматизация деплоя, конфигурации и мониторинга.
