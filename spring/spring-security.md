# Введение в безопасность веб-приложений

## Почему безопасность важна

Веб-приложения доступны по сети и обрабатывают данные пользователей (учётные записи, платежи, персональные данные). Уязвимости ведут к утечкам, мошенничеству, нарушению законодательства (GDPR и др.) и репутационным рискам. Безопасность должна учитываться на этапе проектирования и реализации, а не только при инциденте.

## Основные угрозы веб-приложений

- **Нарушение конфиденциальности** — несанкционированный доступ к данным (утечка паролей, персональных данных).
- **Нарушение целостности** — изменение данных (подмена запросов, инъекции, подделка сессий).
- **Нарушение доступности** — отказ в обслуживании (DDoS, блокировка учётных записей).

## Типичные атаки

| Атака | Суть | Защита |
|-------|------|--------|
| **SQL Injection** | Внедрение SQL-кода через параметры запроса (форма, URL). Злоумышленник может читать/изменять БД или выполнять команды. | Параметризованные запросы (PreparedStatement, JPA), валидация ввода, принцип наименьших привилегий для БД. |
| **XSS (Cross-Site Scripting)** | Внедрение скрипта в страницу (через ввод пользователя). Скрипт выполняется в браузере жертвы, может украсть cookie/сессию. | Экранирование вывода (HTML encoding), Content-Security-Policy, HttpOnly для cookie. |
| **CSRF (Cross-Site Request Forgery)** | Запрос с сайта злоумышленника от имени авторизованного пользователя (браузер отправляет cookie автоматически). | CSRF-токен, SameSite для cookie, проверка Origin/Referer. |
| **Broken Authentication** | Слабые пароли, утечка учётных данных, перебор, отсутствие блокировки, небезопасное хранение паролей. | Сильные хеши (bcrypt, argon2), ограничение попыток входа, MFA, защита сессий. |

Дополнительно: недостаточная авторизация (доступ к чужим ресурсам по подбору ID), небезопасная десериализация, уязвимые зависимости — требуют безопасной разработки и обновлений.

# Authentication vs Authorization

## Authentication (аутентификация)

**Аутентификация** — проверка **личности** субъекта: «кто вы?». Пользователь предъявляет учётные данные (логин/пароль, токен, сертификат); система проверяет их и при успехе считает субъекта идентифицированным. Результат — установление идентичности (например, объект Principal, UserDetails в Spring).

## Authorization (авторизация)

**Авторизация** — проверка **прав доступа**: «что этому субъекту разрешено?». После аутентификации система решает, может ли субъект выполнить действие (доступ к URL, вызов метода, доступ к ресурсу). Проверка ролей (ROLE_ADMIN), прав (permission) или правил (например, «доступ только к своим заказам»).

## Различие

| Аспект | Authentication | Authorization |
|--------|----------------|---------------|
| Вопрос | Кто вы? | Что вам разрешено? |
| Когда | При входе (логин) или при предъявлении токена. | При каждом обращении к защищённому ресурсу. |
| Результат | Установленный субъект (Principal). | Разрешение или отказ в доступе. |

Сначала выполняется аутентификация; затем для каждого защищённого действия — авторизация.

# Основы криптографии

## Три понятия

| Понятие | Суть | Обратимость | Типичное применение |
|---------|------|-------------|----------------------|
| **Encoding (кодирование)** | Представление данных в другом формате (например, бинарные данные как текст). Не предназначено для сокрытия. | Обратимо по правилу кодирования. | Base64 для передачи бинарных данных в JSON/XML, URL. |
| **Hashing (хеширование)** | Односторонняя функция: произвольные данные → фиксированной длины хеш. Одинаковые данные → один хеш. | Необратимо (по замыслу). | Хранение паролей, проверка целостности, подпись. |
| **Encryption (шифрование)** | Преобразование данных с ключом так, что без ключа данные недоступны. | Обратимо при наличии ключа. | Защита данных при хранении и передаче (HTTPS, шифрование полей). |

## Таблица сравнения

| Критерий | Encoding | Hashing | Encryption |
|----------|----------|---------|------------|
| Цель | Удобный формат | Проверка/пароли | Сокрытие содержимого |
| Ключ | Не используется | Не используется (или salt) | Секретный/публичный ключ |
| Восстановление данных | Да (декодирование) | Нет | Да (расшифровка) |
| Один и тот же вывод | Один и тот же ввод | Один и тот же ввод (+ salt) | Один и тот же ввод и ключ |

# Encoding

## Что такое encoding

**Encoding (кодирование)** — преобразование данных из одного представления в другое без цели скрыть содержимое. Например, бинарные данные представляются в виде текстовых символов для передачи в протоколах, работающих только с текстом (JSON, XML, URL). Декодирование восстанавливает исходные данные по правилам того же кодека.

## Зачем используется

- Передача бинарных данных (файлы, подписи) в текстовых полях (JSON, заголовки).
- Безопасное представление байтов в URL (URL encoding) и в данных форм.
- Хранение бинарных данных в системах, поддерживающих только текст.

## Base64

**Base64** — схема кодирования: 3 байта (24 бита) отображаются в 4 символа из набора 64 (A–Z, a–z, 0–9, +, /). Размер данных увеличивается примерно на 33%. Не является шифрованием — декодирование не требует ключа. Используется, например, для вложения изображений в HTML/JSON, для Basic Auth (логин:пароль в заголовке в Base64), для хранения бинарных частей JWT.

Пример (концептуально): `"Hello"` → байты → Base64 → `"SGVsbG8="`. Декодирование по той же схеме возвращает исходные байты.

# Hashing

## Что такое hash

**Хеш (hash)** — результат применения **хеш-функции** к данным произвольной длины. Результат имеет фиксированную длину (например, 256 бит для SHA-256). Одинаковые входные данные всегда дают один и тот же хеш (при одном и том же salt, если он используется).

## Зачем используется hashing

- **Хранение паролей** — пароль не хранится в открытом виде; хранится только хеш (с salt). При входе хешируют введённый пароль и сравнивают с хранимым хешем.
- **Проверка целостности** — хеш файла или сообщения позволяет убедиться, что данные не изменились.
- **Подписи и идентификаторы** — хеш как часть подписи (HMAC) или как идентификатор содержимого.

## Свойства хеш-функций

- **Детерминированность** — один и тот же ввод → один и тот же вывод.
- **Быстрое вычисление** — для больших объёмов данных и проверок.
- **Лавинный эффект** — малое изменение входа сильно меняет хеш.
- **Односторонность** — по хешу практически невозможно восстановить исходные данные (для криптографических хешей).
- **Устойчивость к коллизиям** — сложно найти два разных входа с одинаковым хешем (важно для криптографических хешей).

Для паролей дополнительно используют **salt** и **медленные** функции (bcrypt, argon2), чтобы усложнить перебор.

# Алгоритмы хеширования

## Обзор алгоритмов

| Алгоритм | Тип | Размер выхода | Применение |
|----------|-----|----------------|------------|
| **MD5** | Общего назначения | 128 бит | Устарел; не для паролей и безопасности. |
| **SHA-1** | Общего назначения | 160 бит | Устарел для подписей; не для новых систем. |
| **SHA-256** | Общего назначения | 256 бит | Целостность, подписи; не для паролей (слишком быстрый). |
| **bcrypt** | Адаптивный, для паролей | 192 бит (включая salt) | Рекомендуется для паролей. |
| **scrypt** | Адаптивный, для паролей | Настраивается | Пароли; требует больше памяти. |
| **argon2** | Адаптивный, для паролей | Настраивается | Рекомендуется для новых систем (пароли). |

## Почему MD5 и SHA-1 небезопасны

- **MD5** — коллизии найдены; можно подобрать два разных входа с одним хешем. Не использовать для подписей и паролей.
- **SHA-1** — коллизии теоретически и практически продемонстрированы; не рекомендуется для новых подписей и криптографических сценариев.

Оба к тому же **быстрые** — перебор паролей по радужным таблицам и GPU эффективен. Для паролей нужны **медленные** и **солевые** алгоритмы: bcrypt, scrypt, argon2.

## bcrypt, scrypt, argon2

- **bcrypt** — встроенный salt, настраиваемый cost (число итераций); широко поддерживается (в т.ч. Spring Security). Хороший выбор для паролей.
- **scrypt** — требует много памяти (защита от перебора на GPU/ASIC); настройка сложнее.
- **argon2** — победитель Password Hashing Competition; настраиваемые параметры по времени и памяти. Рекомендуется для новых приложений (Spring Security поддерживает через DelegatingPasswordEncoder).

# Password Storage

## Почему нельзя хранить пароли в открытом виде

При компрометации БД (утечка, бэкап, SQL injection) все пароли становятся известны злоумышленнику. Пользователи часто используют один пароль на нескольких сервисах — утечка одного сайта ставит под удар другие. Законодательство и стандарты (PCI DSS, рекомендации по персональным данным) требуют не хранить пароли в plaintext.

## Как хранить пароли безопасно

- **Не хранить пароль в открытом виде** — только хеш (и при необходимости метаданные: salt, версия алгоритма).
- **Salt (соль)** — случайные данные, уникальные для каждого пароля, добавляемые к паролю перед хешированием. Исключает использование готовых радужных таблиц и одинаковых хешей для одинаковых паролей. В bcrypt/argon2 salt встроен в результат.
- **Адаптивное хеширование** — алгоритм с настраиваемой сложностью (cost factor, итерации, память). Со временем можно увеличивать сложность без смены формата. Примеры: bcrypt (cost), argon2 (время и память).
- **bcrypt** — типичный выбор: встроенный salt, cost factor, устойчивость к перебору. В Spring — BCryptPasswordEncoder.

Пример проверки пароля: при регистрации — `passwordEncoder.encode(rawPassword)` и сохранение хеша в БД; при входе — `passwordEncoder.matches(rawPassword, storedHash)`.

# Encryption

## Симметричное шифрование

Один и тот же **секретный ключ** используется для **шифрования и расшифровки**. Отправитель и получатель должны иметь один ключ. Плюсы: быстро, подходит для больших объёмов. Минусы: безопасная передача ключа (или согласование по протоколу, например Diffie–Hellman).

**AES (Advanced Encryption Standard)** — стандарт симметричного шифрования (128/192/256 бит ключ, блоки 128 бит). Режимы: CBC, GCM (с аутентификацией). Используется в TLS, шифровании дисков, защите данных в покое.

## Асимметричное шифрование

Используются **два ключа**: **приватный** (секретный) и **публичный**. Что зашифровано публичным ключом, расшифровывается только приватным, и наоборот. Публичный ключ можно распространять; приватный хранится у владельца. Плюсы: не нужна предварительная передача общего секрета. Минусы: медленнее симметричного; обычно используют для обмена симметричным ключом или для подписей.

**RSA** — алгоритм с открытым ключом: шифрование, подписи, обмен ключами. Размер ключа обычно 2048–4096 бит. Используется в TLS (обмен ключами), подписи (JWT, сертификаты), шифровании небольших объёмов.

На практике часто комбинируют: асимметрично передают/согласовывают симметричный ключ, дальше данные шифруют симметрично (AES).

# JWT

## Что такое JSON Web Token

**JWT (JSON Web Token)** — компактный способ представления утверждений (claims) между двумя сторонами в формате JSON. Состоит из трёх частей (Header.Payload.Signature), закодированных в Base64URL и объединённых точками. Подписывается (HMAC или RSA) или шифруется (JWE); при подписи получатель может проверить целостность и источник без обращения к серверу, выдавшему токен.

## Зачем используется JWT

- **Stateless-аутентификация** — сервер не хранит сессии; достаточно проверить подпись токена и прочитать claims (user id, roles, expiry). Масштабирование без общего хранилища сессий.
- **Обмен утверждениями** — передача идентификатора пользователя, ролей, scope между сервисами (API Gateway, микросервисы) без повторного запроса к серверу аутентификации при каждом запросе.
- **OAuth2 / OpenID Connect** — access token и id token часто в формате JWT.

## Где применяется

- REST API: клиент после логина получает JWT, отправляет его в заголовке (например, Authorization: Bearer &lt;token&gt;) при каждом запросе.
- Микросервисы: шлюз или сервис проверяет JWT и извлекает пользователя/роли для авторизации.
- Single Page Application (SPA) и мобильные приложения: токен хранится на клиенте (memory, httpOnly cookie — по стратегии), отправляется с запросами.

# Структура JWT

## Три части токена

JWT — строка вида `Header.Payload.Signature`, каждая часть в **Base64URL**.

### Header

Содержит тип токена и алгоритм подписи (или шифрования). Пример: `{"alg":"HS256","typ":"JWT"}`. Алгоритмы: HS256 (симметричная подпись), RS256 (RSA), ES256 (ECDSA) и др.

### Payload (Claims)

**Утверждения (claims)** — данные о субъекте и метаданные. Стандартные (зарезервированные) поля, например:

| Claim | Описание |
|-------|----------|
| **sub** | Subject — идентификатор субъекта (часто user id). |
| **iss** | Issuer — кто выдал токен. |
| **aud** | Audience — для кого предназначен. |
| **exp** | Expiration — время истечения (Unix timestamp). |
| **iat** | Issued At — время выдачи. |
| **nbf** | Not Before — не действует до этого времени. |

Дополнительно можно хранить роли (roles), scope, email и т.д. Не помещать в payload чувствительные данные и пароли — payload при JWS только подписан, но не шифрован (Base64 декодируется любым).

### Signature

Подпись формируется от строки `Base64URL(Header).Base64URL(Payload)` с использованием секретного ключа (HS256) или приватного ключа (RS256). Проверка: повторить вычисление с тем же алгоритмом и ключом; при совпадении подписи токен не изменялся и выдан держателем ключа.

# Как работает JWT

## Последовательность

1. **Пользователь логинится** — отправляет логин и пароль (например, POST /auth/login) по защищённому каналу (HTTPS).
2. **Сервер проверяет данные** — проверяет учётные данные (UserDetailsService, PasswordEncoder); при успехе создаёт JWT с нужными claims (sub, roles, exp, iss и т.д.).
3. **Создаётся JWT** — формируются Header, Payload, подпись; строка `Header.Payload.Signature` возвращается клиенту (в теле ответа или в заголовке).
4. **Клиент хранит токен** — сохраняет в памяти (JS), в localStorage (менее безопасно из-за XSS) или получает в httpOnly cookie (защита от XSS, настройка домена и CORS).
5. **Клиент отправляет токен в каждом запросе** — заголовок `Authorization: Bearer <token>` (или cookie, если токен в cookie). Сервер проверяет подпись, exp, iss/aud при необходимости, извлекает субъекта и права и выполняет авторизацию.

Сервер при этом **не хранит** сессию для этого токена; достаточно криптографической проверки. Отзыв до истечения срока требует чёрного списка токенов или короткого времени жизни токена + refresh token.

# Spring Security

## Что такое Spring Security

**Spring Security** — фреймворк для аутентификации и авторизации в приложениях Spring. Предоставляет защиту от типовых атак (CSRF, session fixation, безопасные заголовки), абстракции для пользователей (UserDetails), паролей (PasswordEncoder), проверки прав (роли, методы, URL) и интеграцию с сервлетами (фильтры), реактивным стеком и OAuth2/JWT.

## Какие задачи решает

- **Аутентификация** — проверка логина/пароля, загрузка пользователя из БД, поддержка JWT, OAuth2, LDAP и др.
- **Авторизация** — ограничение доступа по URL (конфигурация правил), по вызову методов (@PreAuthorize), по данным (ACL, кастомная логика).
- **Защита приложения** — CSRF, CORS, безопасные заголовки (X-Frame-Options, Content-Security-Policy и т.д.), защита сессии.
- **Интеграция** — с Spring MVC (фильтры перед DispatcherServlet), с Spring Data, с микросервисами (Resource Server, JWT).

# Архитектура Spring Security

## Основные компоненты

| Компонент | Назначение |
|-----------|------------|
| **Security Filter Chain** | Цепочка сервлетных фильтров; каждый запрос проходит через неё до DispatcherServlet. Фильтры выполняют аутентификацию, проверку токена, CSRF, авторизацию и т.д. |
| **AuthenticationManager** | Интерфейс: принимает Authentication (учётные данные), возвращает полностью заполненный Authentication (principal, authorities) при успехе или выбрасывает исключение. Делегирует одному или нескольким AuthenticationProvider. |
| **AuthenticationProvider** | Выполняет проверку для конкретного типа учётных данных (например, UsernamePasswordAuthenticationToken). Загружает пользователя (UserDetailsService), проверяет пароль (PasswordEncoder), при успехе создаёт Authentication с principal и authorities. |
| **UserDetailsService** | Загрузка пользователя по имени (username): возвращает UserDetails (логин, пароль, права, флаги). Используется провайдером при аутентификации по логину/паролю. |
| **PasswordEncoder** | Кодирование пароля при регистрации и проверка при входе: matches(rawPassword, encodedPassword). Стандартная реализация для паролей — BCryptPasswordEncoder. |

Схема: запрос → Filter Chain → при необходимости аутентификация (провайдер вызывает UserDetailsService и PasswordEncoder) → сохранение Authentication в SecurityContext → проверка прав (авторизация) → доступ к приложению или 403/401.

# Security Filter Chain

## Как работают фильтры

Spring Security регистрирует цепочку сервлетных фильтров (**SecurityFilterChain**). Каждый HTTP-запрос до попадания в DispatcherServlet проходит через эти фильтры по порядку. Фильтр может: проверить токен/cookie, вызвать AuthenticationManager, записать результат в SecurityContextHolder, проверить CSRF, отклонить запрос (401/403) или передать дальше (chain.doFilter).

## Порядок обработки (типичная цепочка)

1. **Фильтры порядка/логирования** (при наличии).
2. **CorsFilter** — обработка preflight и CORS-заголовков.
3. **CsrfFilter** — для stateful-сессий проверяет CSRF-токен для изменяющих запросов.
4. **UsernamePasswordAuthenticationFilter** (или аналог) — для POST на URL логина извлекает логин/пароль, вызывает AuthenticationManager, при успехе создаёт сессию и записывает Authentication в SecurityContext.
5. **RequestCache, SessionManagement** — восстановление запроса после логина, управление сессией.
6. **FilterSecurityInterceptor** (или аналог) — проверка авторизации: по правилам (URL, роли) решает, разрешить доступ или вернуть 403. Обычно последний в цепочке безопасности.

Для JWT типично добавляют фильтр перед FilterSecurityInterceptor: извлечение токена из заголовка, проверка подписи и срока, создание Authentication и запись в SecurityContext. Тогда UsernamePasswordAuthenticationFilter может использоваться только для эндпоинта логина.

# AuthenticationManager

## Как происходит аутентификация

- Клиент отправляет учётные данные (логин/пароль, токен).
- Соответствующий фильтр создаёт объект **Authentication** (например, UsernamePasswordAuthenticationToken) с credentials и передаёт его в **AuthenticationManager.authenticate(authentication)**.
- AuthenticationManager перебирает зарегистрированные **AuthenticationProvider**; тот, который поддерживает данный тип Authentication, выполняет проверку: загружает пользователя (UserDetailsService), проверяет пароль (PasswordEncoder) или проверяет токен. При успехе провайдер возвращает новый Authentication с заполненными principal и authorities.
- Этот объект сохраняется в **SecurityContextHolder** (и при необходимости в сессии). Дальше FilterSecurityInterceptor и методовая безопасность используют его для авторизации.

При неудаче провайдер выбрасывает исключение (BadCredentialsException и др.); фильтр возвращает 401 или перенаправляет на форму логина.

# UserDetailsService

## Как Spring получает пользователя из БД

**UserDetailsService** — интерфейс с одним методом: **loadUserByUsername(String username)**. Возвращает **UserDetails** (логин, хеш пароля, права, флаги: аккаунт не заблокирован, не истёк и т.д.). Реализацию предоставляет приложение: обычно репозиторий пользователей по имени находит сущность и маппит её в UserDetails (или в класс User из Spring Security).

AuthenticationProvider (например, DaoAuthenticationProvider) при аутентификации по логину/паролю вызывает loadUserByUsername(username), затем passwordEncoder.matches(rawPassword, userDetails.getPassword()). При совпадении пароля создаётся успешный Authentication с principal = userDetails (или сущность пользователя) и authorities из userDetails.getAuthorities().

Пример:

```java
@Service
public class MyUserDetailsService implements UserDetailsService {
    private final UserRepository userRepository;

    @Override
    public UserDetails loadUserByUsername(String username) {
        User user = userRepository.findByEmail(username)
            .orElseThrow(() -> new UsernameNotFoundException(username));
        return new org.springframework.security.core.userdetails.User(
            user.getEmail(), user.getPasswordHash(), user.getAuthorities());
    }
}
```

# PasswordEncoder

## Как Spring кодирует пароли

**PasswordEncoder** — интерфейс: **encode(CharSequence rawPassword)** — для сохранения пароля при регистрации; **matches(CharSequence rawPassword, String encodedPassword)** — для проверки при входе. Spring Security не хранит пароли в открытом виде; в БД хранится только результат encode (с salt внутри, если алгоритм это поддерживает).

При регистрации/смене пароля вызывается encoder.encode(rawPassword); результат сохраняется в БД. При аутентификации провайдер вызывает encoder.matches(rawPassword, storedHash); при true пароль считается верным.

## BCryptPasswordEncoder

**BCryptPasswordEncoder** — реализация на основе bcrypt. Встроенный salt; при каждом encode результат разный. Cost factor (по умолчанию 10) задаёт число итераций. Рекомендуется для новых приложений; для миграции со старых хешей используют **DelegatingPasswordEncoder** с несколькими префиксами (bcrypt, noop, pbkdf2 и т.д.).

Пример конфигурации:

```java
@Bean
public PasswordEncoder passwordEncoder() {
    return new BCryptPasswordEncoder(12);
}
```

Использование при регистрации: `user.setPasswordHash(passwordEncoder.encode(plainPassword));`. В UserDetailsService возвращается этот же хеш в getPassword(); провайдер сам вызывает matches.

# Stateless vs Stateful Security

## Session-based authentication (stateful)

- После успешного логина сервер создаёт **сессию** (идентификатор хранится в cookie, например JSESSIONID). Состояние (кто вошёл) хранится на сервере (в памяти или в хранилище сессий).
- Каждый запрос несёт cookie; сервер по id сессии находит пользователя и проверяет права.
- Плюсы: возможность мгновенно инвалидировать сессию (logout, смена пароля). Минусы: нужен общий или липкий хранилищ сессий при масштабировании; уязвимость к CSRF, если не используется токен/другие меры.

## Token-based authentication (stateless)

- После логина сервер возвращает **токен** (JWT); сервер не хранит сессию. Клиент отправляет токен в заголовке (или cookie). Сервер проверяет подпись и claims и восстанавливает субъекта из токена.
- Плюсы: масштабирование без общего хранилища сессий; удобно для API и микросервисов. Минусы: отзыв до истечения срока сложнее (чёрный список или короткий TTL + refresh token); нужно защищать хранение токена на клиенте (XSS, при cookie — настройка httpOnly, SameSite).

Выбор зависит от требований: SPA/мобильное приложение и API часто используют JWT; классические веб-формы с серверным рендерингом — сессии с защитой от CSRF.

# CSRF

## Что такое CSRF

**CSRF (Cross-Site Request Forgery)** — атака, при которой запрос к вашему сайту инициируется с другого сайта от имени авторизованного пользователя. Браузер автоматически отправляет cookie вашего домена; если аутентификация по cookie (сессия), сервер считает запрос легитимным и выполняет действие (перевод, смена пароля и т.д.).

## Как Spring защищает от CSRF

- **CSRF-токен** — при отдаче формы (или при первом запросе) сервер генерирует случайный токен и сохраняет его в сессии; в форме (или в meta/заголовке) токен передаётся клиенту. При отправке изменяющего запроса (POST, PUT, DELETE) клиент должен отправить этот токен (в теле или заголовке). CsrfFilter сравнивает токен с сессией; при несовпадении или отсутствии — 403.
- Для **stateless API с JWT** в заголовке CSRF часто не используется: токен не отправляется браузером автоматически с другого домена (если не cookie). Если JWT в cookie, нужны те же меры (CSRF-токен или SameSite=Strict/Lax) и осторожность с CORS.
- Дополнительно: проверка заголовка **Origin/Referer** (должен совпадать с вашим доменом) как дополнительный барьер.

# CORS

## Что такое CORS

**CORS (Cross-Origin Resource Sharing)** — механизм браузера, позволяющий серверу явно разрешить запросы с других источников (origin = схема + хост + порт). Браузер по умолчанию блокирует JavaScript-запросы с одного домена к другому (same-origin policy). Заголовки ответа **Access-Control-Allow-Origin**, **Access-Control-Allow-Methods**, **Access-Control-Allow-Headers** и т.д. сообщают браузеру, какие запросы разрешены.

## Зачем нужен

- SPA на одном домене (или поддомене), API на другом — без CORS браузер заблокирует запросы из JS.
- Мобильное приложение или внешний клиент обычно не подчиняются CORS (это ограничение браузера); для API важно корректно настроить CORS, чтобы легитимные веб-клиенты могли обращаться к API.

В Spring Security CORS можно настроить в конфигурации безопасности (cors().configurationSource(...)) или через отдельный CorsFilter; важно не разрешать все origins в production без необходимости.

# Method Security

## Аннотации проверки доступа

Проверка прав на уровне **вызова метода**: перед или после выполнения метода проверяется, что у текущего пользователя есть нужная роль или право.

| Аннотация | Когда выполняется | Типичное использование |
|-----------|-------------------|-------------------------|
| **@PreAuthorize** | До вызова метода. Поддерживает SpEL. | `@PreAuthorize("hasRole('ADMIN')")`, `@PreAuthorize("hasAuthority('user:read')")`, `@PreAuthorize("#id == authentication.principal.id")`. |
| **@PostAuthorize** | После вызова метода; результат доступен в SpEL. | Проверка по возвращённому объекту: `@PostAuthorize("returnObject.owner == authentication.name")`. |
| **@Secured** | До вызова. Только список ролей. | `@Secured("ROLE_ADMIN")`. |

Для работы аннотаций включают глобально **@EnableMethodSecurity** (или в старом стиле @EnableGlobalMethodSecurity) и убеждаются, что в контексте есть настроенная цепочка безопасности (чтобы в SecurityContext был Authentication).

Пример:

```java
@PreAuthorize("hasRole('ADMIN')")
public void deleteUser(Long id) { ... }

@PreAuthorize("#order.userId == authentication.principal.id")
public void updateOrder(Order order) { ... }
```

# OAuth2

## Что такое OAuth2

**OAuth 2.0** — протокол **авторизации** (делегирования доступа). Позволяет клиенту (приложению) получить ограниченный доступ к ресурсам пользователя на другом сервере (например, доступ к профилю Google) без передачи пароля клиенту. Роли: **Resource Owner** (пользователь), **Client** (приложение), **Authorization Server** (выдаёт токены), **Resource Server** (API, защищённый токенами). Клиент получает **access token** и опционально **refresh token**; доступ к API — по access token.

## Зачем используется

- **Вход через соцсети/провайдеры** — «Login with Google/GitHub»: Authorization Server провайдера выдаёт токен, ваше приложение проверяет его и создаёт свою сессию или JWT.
- **Защита API** — микросервисы как Resource Server проверяют access token (JWT или opaque), выданный вашим или внешним Authorization Server.
- **Делегирование доступа** — стороннее приложение с согласия пользователя получает доступ к вашему API по токену.

Spring Security поддерживает OAuth2: клиент (OAuth2Client), Resource Server (проверка JWT или запрос к introspection endpoint), Authorization Server (Spring Authorization Server или интеграция с Keycloak и др.).

# Best Practices безопасности

## Хранение паролей

- Хранить только **хеш** с **salt** (bcrypt, argon2); никогда plaintext.
- Использовать **адаптивные** алгоритмы и увеличивать cost со временем.
- Не использовать MD5/SHA-1 для паролей.

## Защита API

- **HTTPS** для всех запросов; токены и пароли не передавать по незашифрованному каналу.
- **JWT**: короткий срок жизни access token; refresh token хранить безопасно и отзывать при logout. Проверять подпись, exp, iss/aud.
- **Ограничение частоты запросов** (rate limiting) для логина и чувствительных эндпоинтов.
- **Валидация и санитизация ввода**; параметризованные запросы к БД.

## Ограничение доступа

- **Принцип наименьших привилегий** — роли и права только по необходимости.
- Проверка авторизации на **каждом** защищённом эндпоинте и при доступе к ресурсам (не только на уровне URL, но и «доступ только к своим данным»).
- Method Security (@PreAuthorize) для критичных операций.

## Использование HTTPS

- TLS для всего трафика; редирект HTTP → HTTPS.
- Безопасные cookie: **Secure**, **HttpOnly**; при кросс-доменной работе — **SameSite**.
- HSTS-заголовок для принудительного использования HTTPS в браузере.

# Краткий конспект для собеседования

## Authentication vs Authorization

- **Authentication** — «кто вы?»: проверка учётных данных (логин/пароль, токен). Результат — установленный субъект (Principal).
- **Authorization** — «что вам разрешено?»: проверка прав на действие или ресурс после аутентификации.

## Hashing vs Encryption

- **Hashing** — односторонняя функция; необратимо; для паролей и проверки целостности. С salt и медленными алгоритмами (bcrypt, argon2).
- **Encryption** — обратимо с ключом; для сокрытия данных. Симметричное (AES) и асимметричное (RSA).

## JWT структура

- **Header** — alg, typ (Base64URL).
- **Payload** — claims: sub, exp, iat, roles и др. (Base64URL).
- **Signature** — подпись от Header.Payload (HMAC или RSA). Проверка подписи без обращения к серверу.

## Security Filter Chain

- Цепочка сервлетных фильтров до DispatcherServlet. Порядок: CORS → CSRF → аутентификация (логин/JWT) → сохранение в SecurityContext → авторизация (FilterSecurityInterceptor). Каждый запрос проходит через цепочку.

## PasswordEncoder

- **encode(rawPassword)** — при регистрации/смене пароля; результат хранить в БД.
- **matches(rawPassword, encodedPassword)** — при входе. Рекомендуется **BCryptPasswordEncoder**; для миграции — DelegatingPasswordEncoder.
