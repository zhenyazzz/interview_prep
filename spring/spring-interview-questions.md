# Основы Spring

## Что такое Spring Framework?

**Определение.** Spring Framework — платформа для построения Java-приложений, которая даёт инфраструктуру для управления объектами (IoC), внедрения зависимостей (DI), конфигурации, работы с данными, веб-слоя и безопасности.

**Зачем.** Уменьшает связность кода, выносит конфигурацию и создание объектов в контейнер, упрощает тестирование (подмена зависимостей) и интеграцию библиотек.

**Как работает.** Контейнер (ApplicationContext) по конфигурации (XML, аннотации, Java-конфиг) создаёт объекты (beans), подставляет зависимости и управляет жизненным циклом. Приложение получает готовые бины через DI, а не создаёт их через `new`.

**Пример.** Вместо `new OrderService(repository)` класс объявляет зависимость в конструкторе; Spring создаёт репозиторий и сервис и передаёт репозиторий в сервис.

---

## Какие основные модули Spring существуют?

| Модуль | Назначение |
|--------|------------|
| **Spring Core** | IoC, DI, утилиты (Resource, Environment). |
| **Spring Context** | ApplicationContext, события, профили, аннотационная конфигурация. |
| **Spring Beans** | BeanFactory, BeanDefinition, жизненный цикл bean. |
| **Spring AOP** | Прокси, аспекты, сквозная логика (транзакции, безопасность). |
| **Spring Web / MVC** | DispatcherServlet, контроллеры, REST, маппинг запросов. |
| **Spring Data** | Репозитории, JPA, упрощение доступа к данным. |
| **Spring Security** | Аутентификация, авторизация, защита от типовых атак. |

Остальные (JDBC, Tx, Messaging и др.) строятся поверх Core и Context.

---

## Чем Spring Boot отличается от Spring Framework?

**Spring Framework** — ядро: IoC, DI, MVC, Data и т.д. Конфигурация и сборка приложения задаются вручную (контекст, зависимости, сервер).

**Spring Boot** — надстройка над Spring: автоконфигурация по classpath, стартеры (наборы зависимостей), встроенный сервер, конвенции. Минимум кода для запуска приложения; конфигурация по умолчанию из коробки.

**Кратко:** Framework — «движок»; Boot — быстрый старт и удобная упаковка приложения (в т.ч. executable JAR).

# IoC и Dependency Injection

## Что такое Inversion of Control?

**Определение.** IoC — подход, при котором **управление созданием и жизненным циклом объектов передаётся контейнеру**, а не коду приложения.

**Зачем.** Класс не создаёт зависимости сам (`new`), а получает их извне. Снижается связность, проще подмена реализаций и тестирование.

**Как работает.** Контейнер знает, какие классы создавать и как связывать (из конфигурации и аннотаций). Он создаёт объекты, подставляет зависимости и отдаёт готовые бины. «Контроль» инвертирован: не приложение решает, что создавать, а контейнер.

---

## Что такое Dependency Injection?

**Определение.** DI — способ реализации IoC: **зависимости передаются** объекту извне (контейнером), а не создаются внутри класса.

**Зачем.** Явные зависимости, возможность подставлять разные реализации (в т.ч. моки в тестах) без изменения кода класса.

**Как работает.** Класс объявляет зависимости (конструктор, сеттер, поле); контейнер при создании bean разрешает другие beans по типу/имени и подставляет их. В Spring — через конструктор, setter или поле с @Autowired.

---

## Какие типы DI существуют?

| Тип | Описание | В Spring |
|-----|----------|----------|
| **Constructor** | Зависимости через конструктор. | Рекомендуется: явно, можно `final`, удобно для тестов. |
| **Setter** | Зависимости через setter-методы. | Опциональные или изменяемые зависимости. |
| **Field** | Инжект в поле через рефлексию. | @Autowired на поле; не рекомендуется (сложнее тесты, нет final). |

---

## Почему constructor injection считается лучшей практикой?

- **Неизменяемость** — поля можно сделать `final`, зависимости не меняются после создания.
- **Явный контракт** — все обязательные зависимости видны в конструкторе; нельзя создать bean без них.
- **Тестирование** — в тестах зависимости передаются в конструктор без Spring.
- **Циклические зависимости** — в ряде случаев Spring может разрешить цикл при constructor injection (хотя циклы лучше избегать).

# Spring Beans

## Что такое Spring Bean?

**Определение.** Bean — объект, **создаваемый и управляемый IoC-контейнером** Spring. Описывается метаданными (BeanDefinition): класс, scope, зависимости и т.д.

**Зачем.** Единая точка создания и конфигурации; один экземпляр (или новый при каждом запросе — по scope) переиспользуется; зависимости подставляются контейнером.

**Как работает.** Контейнер по BeanDefinition создаёт экземпляр, внедряет зависимости, применяет BeanPostProcessor (например, для AOP), вызывает инициализацию (@PostConstruct). Singleton хранится в контексте и возвращается при повторных запросах.

---

## Что такое Bean Scope?

**Определение.** Scope задаёт **правило создания и жизни** экземпляра bean: один на контейнер, новый при каждом запросе или на запрос/сессию (в веб-контексте).

**Зачем.** Контроль над количеством экземпляров и временем жизни; экономия памяти (singleton) или изоляция (prototype, request).

**Как работает.** При запросе bean контейнер смотрит scope: для singleton возвращает кэшированный экземпляр; для prototype создаёт новый; для request/session привязывает к текущему запросу/сессии.

---

## Какие scopes существуют?

| Scope | Описание |
|-------|----------|
| **singleton** | Один экземпляр на контейнер (по умолчанию). |
| **prototype** | Новый экземпляр при каждом запросе bean. |
| **request** | В веб-контексте — один на HTTP-запрос. |
| **session** | В веб-контексте — один на HTTP-сессию. |
| **application** | В веб-контексте — один на ServletContext. |

Задаётся аннотацией @Scope("prototype") или аналогом.

---

## Чем singleton scope отличается от классического Singleton?

**Singleton (паттерн)** — один экземпляр на **всё приложение (ClassLoader)**; создаётся самим классом (статическое поле или enum).

**Spring singleton scope** — один экземпляр **на контейнер (ApplicationContext)**. Может быть несколько контекстов — тогда несколько «singleton» bean. Создание и хранение ведёт контейнер, а не класс. Это «контейнерный» синглтон, а не глобальный синглтон процесса.

# Bean Lifecycle

## Какой жизненный цикл Spring Bean?

1. **BeanDefinition** — регистрация метаданных из конфигурации/scan.
2. **Instantiation** — создание объекта (конструктор или фабрика).
3. **Dependency injection** — подстановка зависимостей.
4. **BeanPostProcessor.postProcessBeforeInitialization** — логика до инициализации (например, часть @Autowired).
5. **Инициализация** — @PostConstruct → InitializingBean.afterPropertiesSet() → init-method.
6. **BeanPostProcessor.postProcessAfterInitialization** — часто создаётся прокси (AOP); контейнер хранит то, что вернул метод.
7. **Bean готов** — использование; для singleton — в кэше.
8. **Уничтожение** (при закрытии контекста) — @PreDestroy → DisposableBean.destroy() → destroy-method.

---

## Что делает BeanPostProcessor?

**Определение.** Интерфейс контейнера: возможность **изменить или заменить** bean до и после инициализации.

**Зачем.** Добавить логику ко всем создаваемым beans (например, внедрение @Autowired, создание прокси для AOP, валидация) без изменения ядра контейнера.

**Как работает.** Контейнер вызывает **postProcessBeforeInitialization** (после DI, до @PostConstruct) и **postProcessAfterInitialization** (после инициализации). Метод может вернуть другой объект — контейнер будет работать с ним (так подставляется прокси для @Transactional).

---

## Для чего используются @PostConstruct и @PreDestroy?

**@PostConstruct** — метод вызывается **после** внедрения зависимостей и **до** использования bean. Удобно для инициализации (подключения, загрузка кэша).

**@PreDestroy** — метод вызывается **при уничтожении** bean (закрытие контекста). Удобно для освобождения ресурсов (закрытие соединений, отмена подписок).

Оба из `javax.annotation`; контейнер вызывает их автоматически в рамках жизненного цикла.

---

# ApplicationContext

## Что такое ApplicationContext?

**Определение.** ApplicationContext — расширение BeanFactory и **основной способ** работы с IoC-контейнером в приложении: создание и выдача bean плюс загрузка конфигурации, события, ресурсы, профили.

**Зачем.** Один объект даёт доступ к бинам, конфигурации, событиям и ресурсам; не нужно вручную поднимать контекст из XML — в Spring Boot контекст создаётся при запуске.

**Как работает.** Реализации (например, AnnotationConfigApplicationContext) загружают конфигурацию (@Configuration, component scan), регистрируют BeanDefinition, создают bean и кэшируют singleton. Метод getBean() возвращает bean по имени или типу.

---

## Чем ApplicationContext отличается от BeanFactory?

| Критерий | BeanFactory | ApplicationContext |
|----------|-------------|---------------------|
| Создание/выдача bean | Да | Да (наследует) |
| Загрузка конфигурации (XML, @Configuration) | Нет | Да |
| События (ApplicationEvent) | Нет | Да |
| Ресурсы (Resource), профили | Ограниченно | Полноценно |
| Типичное использование | Низкоуровневый API | Приложения, Spring Boot |

BeanFactory — минимальный контракт «фабрики» bean; ApplicationContext — полноценный контекст приложения. В коде почти всегда используют ApplicationContext.

# Spring Boot

## Что такое Spring Boot?

**Определение.** Надстройка над Spring Framework: автоконфигурация, стартеры зависимостей, встроенный сервер и «production-ready» функции (Actuator).

**Зачем.** Быстро поднять приложение с минимумом конфигурации; один исполняемый JAR; согласованные версии зависимостей (BOM).

**Как работает.** При старте поднимается ApplicationContext; подключаются автоконфигурации из META-INF (AutoConfiguration.imports / spring.factories) с учётом условий (@ConditionalOnClass и т.д.). Стартеры подтягивают библиотеки и включают нужную автоконфигурацию. Встроенный Tomcat/Jetty запускается из того же процесса.

---

## Что делает аннотация @SpringBootApplication?

**Составная аннотация** из трёх частей:

1. **@SpringBootConfiguration** — класс считается конфигурацией (эквивалент @Configuration).
2. **@EnableAutoConfiguration** — включение автоконфигурации: загрузка классов из AutoConfiguration.imports / spring.factories и применение при выполнении условий.
3. **@ComponentScan** — сканирование компонентов от пакета этого класса (и подпакетов); регистрация @Component, @Service, @Repository, @Controller.

Достаточно одной аннотации на класс с main, чтобы поднять контекст и отсканировать компоненты.

---

## Что такое auto-configuration?

**Определение.** Механизм Spring Boot: **автоматическая** регистрация бинов и настройка приложения по classpath, свойствам и отсутствию своих бинов.

**Зачем.** Не писать вручную конфигурацию DataSource, сервера, Jackson и т.д.; разумные значения по умолчанию; переопределение только при необходимости.

**Как работает.** Классы с @Configuration и @Conditional* перечислены в JAR (AutoConfiguration.imports). При старте контекста они загружаются; проверяются условия (@ConditionalOnClass, @ConditionalOnMissingBean и т.д.); при выполнении регистрируются бины. Так появляются, например, Tomcat и DispatcherServlet без кода в приложении.

---

## Что такое Spring Boot starters?

**Определение.** Стартеры — артефакты с **набором транзитивных зависимостей** для типовой задачи (веб, JPA, безопасность). Имя обычно `spring-boot-starter-*`.

**Зачем.** Одна зависимость подтягивает совместимые библиотеки и подключает автоконфигурацию; не подбирать версии вручную.

**Примеры.** spring-boot-starter-web (MVC, Tomcat, Jackson), spring-boot-starter-data-jpa (JPA, Hibernate), spring-boot-starter-security, spring-boot-starter-test (JUnit, Mockito, MockMvc).

# Spring MVC

## Как работает DispatcherServlet?

**Определение.** Front Controller: **единственная точка входа** для HTTP-запросов в Spring MVC. Сервлет, маппинг обычно на `/`.

**Зачем.** Один компонент принимает все запросы и делегирует поиск обработчика, вызов контроллера и формирование ответа.

**Как работает.** Получает запрос → обращается к **HandlerMapping** (какой контроллер и метод обработают) → **HandlerAdapter** разрешает аргументы и вызывает метод → по возвращаемому значению либо **ViewResolver** (HTML), либо **HttpMessageConverter** (JSON) → ответ клиенту.

---

## Как происходит обработка HTTP-запроса в Spring MVC?

1. Запрос попадает в **DispatcherServlet**.
2. **HandlerMapping** возвращает handler (контроллер + метод) для данного URL и метода HTTP.
3. **HandlerAdapter** подставляет аргументы (@RequestParam, @RequestBody, @PathVariable и т.д.) и вызывает метод контроллера.
4. Контроллер возвращает объект или имя view.
5. Если объект (REST) — **HttpMessageConverter** сериализует в JSON и пишет в ответ. Если имя view — **ViewResolver** находит шаблон и рендерит HTML.
6. Ответ отправляется клиенту.

---

## Что такое @RestController?

**Определение.** @RestController = @Controller + @ResponseBody на уровне класса: все методы считаются возвращающими **тело ответа**, а не имя view.

**Зачем.** Удобно для REST API: возвращаемый объект сериализуется в JSON (или другой формат) через MessageConverter.

**Как работает.** HandlerAdapter обрабатывает возвращаемое значение как тело ответа; выбирается конвертер по типу и Accept; результат записывается в HttpServletResponse. ViewResolver не используется.

# Spring Data JPA

## Что такое JPA?

**Определение.** Java Persistence API — **спецификация** (интерфейсы и правила) для ORM в Java: маппинг сущностей, EntityManager, JPQL.

**Зачем.** Единый стандарт; смена провайдера (Hibernate, EclipseLink) без переписывания кода приложения.

**Как работает.** Приложение использует API JPA (EntityManager, аннотации @Entity, @Id и т.д.); реализацию даёт провайдер (чаще всего Hibernate), который генерирует SQL и управляет persistence context.

---

## Что такое Hibernate?

**Определение.** Популярная **реализация JPA** (провайдер): интерпретирует JPA и свои расширения, генерирует SQL, управляет кэшем и жизненным циклом сущностей.

**Зачем.** ORM «из коробки»: маппинг объект–таблица, запросы (JPQL, Criteria), кэш, dirty checking.

**Как работает.** Реализует EntityManager, EntityManagerFactory; при persist/find/merge выполняет SQL через JDBC; хранит сущности в persistence context (кэш первого уровня), при flush синхронизирует с БД.

---

## Что такое Persistence Context?

**Определение.** Кэш сущностей, которыми управляет **один** EntityManager в рамках транзакции (или сессии). Сущности в нём в состоянии **managed**.

**Зачем.** Отслеживание изменений (dirty checking), единая точка для flush, избежание дублирования загруженных сущностей в одной транзакции.

**Как работает.** При find/persist сущность попадает в контекст. Изменения полей отслеживаются; при flush (или commit) генерируются UPDATE для изменённых сущностей. При закрытии контекста сущности становятся detached. В Spring обычно один контекст на транзакцию.

---

## Что такое Dirty Checking?

**Определение.** Механизм JPA/Hibernate: для сущностей в состоянии **managed** провайдер отслеживает изменения полей; при **flush** генерирует UPDATE для изменившихся сущностей.

**Зачем.** Не вызывать update вручную — достаточно изменить поле у загруженной сущности в той же транзакции.

**Как работает.** Провайдер хранит снимок состояния при загрузке (или после последнего flush). При flush сравнивает текущие значения со снимком; при отличии выполняет UPDATE. Работает только для managed-сущностей; для detached нужен merge.

---

# Транзакции

## Что делает аннотация @Transactional?

**Определение.** Указывает, что метод (или все методы класса) должен выполняться в **транзакции**: при входе открывается транзакция (или используется текущая), при выходе без исключения — commit, при исключении — rollback.

**Зачем.** Декларативное управление транзакциями без ручного открытия/закрытия и обработки исключений; границы транзакции задают и границы persistence context в JPA.

**Как работает.** Прокси (AOP) перехватывает вызов метода: перед вызовом — начало транзакции (или присоединение к существующей по propagation), после вызова — commit; при исключении — rollback. Вызовы «изнутри» того же bean (this.method()) не идут через прокси — транзакция не распространяется на вложенный вызов без прокси.

---

## Что такое propagation?

**Определение.** Propagation задаёт **поведение** при наличии уже открытой транзакции: участвовать в ней или создать новую.

**Зачем.** Контроль границ транзакций при вызове одного @Transactional-метода из другого (одна общая транзакция или отдельная вложенная).

**Как работает.** Перед выполнением метода менеджер транзакций смотрит на текущую транзакцию и выбранный режим (REQUIRED, REQUIRES_NEW и т.д.) и решает: использовать текущую, приостановить и создать новую или выбросить исключение.

---

## Какие propagation типы существуют?

| Тип | Поведение |
|-----|-----------|
| **REQUIRED** (по умолчанию) | Участвовать в текущей транзакции или создать новую. |
| **REQUIRES_NEW** | Всегда новая транзакция; текущая приостанавливается. |
| **SUPPORTS** | С транзакцией — участвовать; без — выполнять без транзакции. |
| **NOT_SUPPORTED** | Выполнять без транзакции; текущую приостановить. |
| **MANDATORY** | Требуется текущая транзакция; иначе исключение. |
| **NEVER** | Транзакции быть не должно; иначе исключение. |
| **NESTED** | Вложенная транзакция (savepoint) внутри текущей. |

---

## Что такое isolation levels?

**Определение.** Уровень изоляции транзакции определяет, какие **побочные эффекты** других транзакций видны: грязное чтение, неповторяемое чтение, фантомное чтение.

**Уровни.** READ_UNCOMMITTED, READ_COMMITTED (часто по умолчанию), REPEATABLE_READ, SERIALIZABLE. Задаётся в @Transactional(isolation = Isolation.REPEATABLE_READ). Чем выше изоляция, тем меньше аномалий, но больше блокировок и ниже параллельность.

---

# N+1 Problem

## Что такое N+1 проблема?

**Определение.** Ситуация, когда для выборки **N** корневых сущностей выполняется **1** запрос на саму выборку и **N** дополнительных запросов к связи (например, LAZY-коллекция у каждой из N сущностей). Итого 1 + N запросов.

**Зачем знать.** Резко ухудшает производительность; типичная ошибка при использовании JPA с LAZY без явной загрузки связей.

**Как возникает.** Загружаются сущности (findAll или по критерию); у каждой есть LAZY-коллекция. При обращении к коллекции (в цикле или при сериализации) для каждой сущности выполняется отдельный SELECT.

---

## Как её обнаружить?

- Включить логирование SQL (Hibernate: show_sql) и увидеть один основной запрос и множество однотипных запросов к одной связи.
- Профилирование БД: много запросов на один HTTP-запрос.
- Мониторинг метрик (количество запросов к БД на запрос).

---

## Как её решить?

- **JOIN FETCH** в JPQL: загрузить связь в одном запросе (`SELECT o FROM Order o JOIN FETCH o.items`). Для коллекций часто нужен DISTINCT.
- **@EntityGraph**: указать атрибуты для eager-загрузки в данном запросе.
- **@BatchSize** на связи: подгрузка коллекций пачками (уменьшает число запросов, но не до одного).

Предпочтительно JOIN FETCH или @EntityGraph — один запрос вместо N+1.

---

# Spring Security

## Что такое Spring Security?

**Определение.** Фреймворк для **аутентификации** и **авторизации** в приложениях Spring: проверка учётных данных, проверка прав доступа, защита от CSRF, CORS и типовых атак.

**Зачем.** Единая точка настройки входа, ролей, защиты URL и методов; интеграция с JWT, OAuth2, сессиями.

**Как работает.** Цепочка фильтров перехватывает запросы: проверка токена/cookie, загрузка пользователя (UserDetailsService), проверка пароля (PasswordEncoder), запись Authentication в SecurityContext; FilterSecurityInterceptor проверяет права по правилам; при успехе запрос идёт в приложение, иначе 401/403.

---

## Чем authentication отличается от authorization?

**Authentication (аутентификация)** — проверка **личности**: «кто вы?» (логин/пароль, токен). Результат — установленный субъект (Principal).

**Authorization (авторизация)** — проверка **прав**: «что вам разрешено?» (роли, права на ресурс). Выполняется после аутентификации при обращении к защищённому ресурсу.

Сначала определяем, кто пользователь; затем — разрешено ли ему действие.

---

## Как работает JWT?

1. Пользователь логинится (логин/пароль).
2. Сервер проверяет данные и создаёт JWT (claims: sub, roles, exp и т.д.), подписывает секретом или ключом.
3. Токен возвращается клиенту (в теле ответа или в cookie).
4. Клиент отправляет токен в заголовке (Authorization: Bearer &lt;token&gt;) при каждом запросе.
5. Сервер проверяет подпись и срок (exp), извлекает claims и строит контекст безопасности (Principal). Обращения к БД для проверки сессии не требуется (stateless).

---

## Из чего состоит JWT?

Три части в формате **Header.Payload.Signature** (каждая в Base64URL):

- **Header** — тип (typ: JWT) и алгоритм подписи (alg: HS256/RS256).
- **Payload** — утверждения (claims): sub (кто), exp (истечение), iat (время выдачи), roles и др. Не шифруется при JWS — только подпись.
- **Signature** — подпись от строки Header.Payload; проверка гарантирует целостность и то, что токен выдан держателем ключа.

# Микросервисы

## Что такое микросервисы?

**Определение.** Архитектурный подход: приложение как набор **независимых сервисов**; у каждого своя зона ответственности, свой деплой и, как правило, своя БД; взаимодействие по сети (HTTP или сообщения).

**Зачем.** Независимые релизы, изоляция отказов, выборочное масштабирование, автономия команд.

**Как работает.** Каждый сервис — отдельное приложение; общение через API или события; обнаружение через Service Discovery; единая точка входа — API Gateway.

---

## Что такое API Gateway?

**Определение.** Единая **точка входа** для клиентов: все запросы идут на шлюз; шлюз маршрутизирует их к микросервисам и выполняет сквозные функции.

**Зачем.** Скрыть внутреннюю топологию; централизованная аутентификация, rate limiting, логирование; один базовый URL для клиента.

**Как работает.** Клиент обращается к шлюзу по URL; шлюз по правилам (path, заголовки) выбирает целевой сервис (по имени из discovery при lb://service-name), добавляет/проверяет заголовки и проксирует запрос. Ответ возвращается клиенту. В Spring — Spring Cloud Gateway.

---

## Что такое Service Discovery?

**Определение.** Механизм **обнаружения инстансов сервисов по имени**: реестр хранит «имя сервиса → список host:port»; сервисы регистрируются при старте; клиенты запрашивают список и выбирают инстанс.

**Зачем.** В облаке адреса динамические; жёсткие URL в конфиге не подходят. Discovery даёт актуальный список инстансов для балансировки и отказоустойчивости.

**Как работает.** Сервис при старте регистрируется в реестре (Eureka, Consul); периодически шлёт heartbeat. Клиент (или шлюз) запрашивает у реестра список по имени и вызывает выбранный инстанс (client-side или через прокси). В Spring Cloud — Eureka, интеграция с OpenFeign и LoadBalancer.

# Частые вопросы на собеседовании

## Почему используется Spring Boot?

- Быстрый старт приложения с минимумом кода и конфигурации.
- Автоконфигурация и стартеры — не нужно вручную собирать контекст и зависимости.
- Встроенный сервер и один исполняемый JAR — удобно для контейнеров и облака.
- Совместимые версии зависимостей (BOM), Actuator для мониторинга.

---

## Что такое Dependency Injection?

- Передача зависимостей объекту **извне** (контейнером), а не создание их внутри класса.
- Снижает связность, упрощает тестирование и смену реализаций.
- В Spring — через конструктор (рекомендуется), setter или поле с @Autowired.

---

## Что такое @Transactional?

- Метод выполняется в **транзакции**: при нормальном выходе — commit, при исключении — rollback.
- Границы транзакции задают и границы persistence context в JPA.
- Реализуется через AOP-прокси; можно задавать propagation, isolation, readOnly, timeout.

---

## Как работает JWT?

- После успешного логина сервер создаёт токен (claims + подпись) и отдаёт клиенту.
- Клиент хранит токен и отправляет в заголовке при каждом запросе.
- Сервер проверяет подпись и exp, извлекает субъекта и права — без обращения к БД (stateless).

---

## Что такое N+1 проблема?

- Один запрос на выборку N сущностей и N дополнительных запросов при обращении к LAZY-связям у каждой.
- Решение: явная загрузка связи в одном запросе (JOIN FETCH, @EntityGraph) или batch size.

---

# Быстрое повторение перед собеседованием

## Cheat Sheet

- **Spring** — платформа с IoC/DI, контекст управляет объектами (beans).
- **IoC** — контроль создания/жизни объектов у контейнера; **DI** — зависимости передаются извне (конструктор лучше).
- **Bean** — объект, создаваемый контейнером; **scope** — правило жизни (singleton, prototype, request, session).
- **Жизненный цикл** — создание → DI → BeanPostProcessor (до/после инициализации) → @PostConstruct → (прокси) → готов; при shutdown — @PreDestroy.
- **ApplicationContext** — контекст приложения (бины + конфигурация, события, ресурсы); расширяет BeanFactory.
- **Spring Boot** — автоконфигурация, стартеры, встроенный сервер; @SpringBootApplication = Configuration + EnableAutoConfiguration + ComponentScan.
- **DispatcherServlet** — Front Controller в MVC; HandlerMapping → HandlerAdapter → контроллер → ViewResolver или MessageConverter.
- **JPA** — спецификация ORM; **Hibernate** — провайдер. **Persistence Context** — кэш managed-сущностей; **Dirty Checking** — автоматический UPDATE при изменении managed-сущности.
- **@Transactional** — метод в транзакции; **propagation** — поведение при существующей транзакции (REQUIRED, REQUIRES_NEW и др.); **isolation** — уровень изоляции.
- **N+1** — 1 запрос + N запросов к LAZY; решать через JOIN FETCH или @EntityGraph.
- **Spring Security** — аутентификация (кто вы) и авторизация (что разрешено); цепочка фильтров, UserDetailsService, PasswordEncoder; **JWT** — Header.Payload.Signature, stateless.
- **Микросервисы** — независимые сервисы, связь по сети; **API Gateway** — единая точка входа; **Service Discovery** — реестр инстансов по имени (Eureka).
