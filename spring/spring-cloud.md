# Введение в Spring Cloud

## Что такое Spring Cloud

**Spring Cloud** — набор библиотек и решений для построения **распределённых систем** и **микросервисов** в экосистеме Spring. Предоставляет готовые паттерны: обнаружение сервисов (Service Discovery), централизованная конфигурация, API Gateway, вызовы между сервисами (REST-клиент с балансировкой), устойчивость к сбоям (circuit breaker, retry), распределённая трассировка и интеграция с очередями сообщений. Строится поверх Spring Boot и использует его автоконфигурацию и конвенции.

## Зачем он нужен

- **Единый стек** — все компоненты интегрированы с Spring (бины, конфигурация, профили). Разработчик остаётся в рамках знакомой модели.
- **Готовые реализации** — не нужно с нуля писать реестр сервисов, шлюз или клиент с балансировкой; достаточно зависимостей и настроек.
- **Совместимость с облаком** — удобно разворачивать в Kubernetes, облачных платформах; часть компонентов (например, discovery) может заменяться возможностями платформы.

## Какие проблемы микросервисов решает

- **Обнаружение сервисов** — динамические адреса инстансов; Eureka (или Consul, Kubernetes) даёт реестр и клиентское обнаружение.
- **Конфигурация** — централизованное хранение и обновление настроек (Spring Cloud Config).
- **Единая точка входа** — маршрутизация, аутентификация, лимиты (Spring Cloud Gateway).
- **Вызовы между сервисами** — декларативный REST-клиент (OpenFeign) с подстановкой адреса из реестра и балансировкой.
- **Устойчивость** — circuit breaker, retry, таймауты (Resilience4j).
- **Наблюдаемость** — трассировка (Micrometer Tracing, Zipkin), метрики, логи.

# Проблемы микросервисной архитектуры

## Основные проблемы

| Проблема | Суть | Решение в Spring Cloud |
|----------|------|-------------------------|
| **Service discovery** | Инстансы меняют IP/порт; жёсткие адреса в конфиге не подходят. | Eureka, Consul; клиент получает список инстансов по имени. |
| **Configuration management** | Много сервисов и инстансов; изменение конфига без пересборки и единого источника. | Spring Cloud Config (Config Server + Git). |
| **Inter-service communication** | Вызовы по HTTP; нужен клиент, выбор инстанса, retry. | OpenFeign + LoadBalancer + discovery. |
| **Fault tolerance** | Отказы и задержки одного сервиса не должны ронять вызывающих. | Resilience4j (circuit breaker, retry, timeout). |
| **Distributed tracing** | Один запрос проходит через несколько сервисов; нужен общий идентификатор и сбор span'ов. | Micrometer Tracing, Zipkin/Jaeger. |
| **Monitoring** | Метрики и здоровье множества инстансов и сервисов. | Actuator, Prometheus, Grafana; health от discovery. |

Без этих механизмов микросервисная система сложна в эксплуатации и отладке; Spring Cloud предлагает согласованный набор решений в рамках Spring.

# Архитектура Spring Cloud

## Экосистема Spring Cloud

Spring Cloud — это **набор подпроектов** (модулей), каждый решает свою задачу. Версии согласованы через **release train** (например, 2023.0.x); совместимость с версией Spring Boot задаётся в документации. Модули подключаются зависимостями к Spring Boot-приложению.

## Основные компоненты

| Компонент | Назначение |
|-----------|------------|
| **Spring Cloud Netflix (частично)** | Eureka (discovery), ранее Hystrix (снят с разработки). Ribbon вытеснен Spring Cloud LoadBalancer. |
| **Spring Cloud Gateway** | API Gateway на WebFlux: маршрутизация, фильтры, интеграция с discovery и безопасностью. |
| **Spring Cloud Config** | Config Server (источник конфигурации из Git, файлов, Vault) и Config Client в сервисах. |
| **Spring Cloud OpenFeign** | Декларативный REST-клиент: интерфейс с аннотациями, интеграция с LoadBalancer и discovery. |
| **Spring Cloud LoadBalancer** | Клиентская балансировка: выбор инстанса из списка (round-robin и др.) при вызове по имени сервиса. |
| **Spring Cloud Stream** | Абстракция над брокерами сообщений (RabbitMQ, Kafka): привязки, каналы, упрощение event-driven сценариев. |

Дополнительно: Spring Cloud Contract (контракты API), Spring Cloud Sleuth (устаревший, заменён Micrometer Tracing), интеграция с Kubernetes (Discovery, Config).

Схема (упрощённо):

```
  [Клиент] ──► [API Gateway] ──► [Eureka] ◄── регистрация сервисов
       │              │
       │              ├──► [Service A] ──► [Config Server] ◄── Git
       │              ├──► [Service B] ◄── OpenFeign + LoadBalancer
       │              └──► [Service C]
       │
       └── трассировка (Zipkin), метрики (Prometheus/Grafana)
```

# Service Discovery

## Зачем нужен Service Discovery

В облаке и оркестраторах инстансы сервисов **создаются и удаляются** динамически, меняются IP и порты. Жёстко прописанные адреса в конфигурации делают систему хрупкой и не позволяют масштабировать и восстанавливаться без ручного изменения конфигов. **Service Discovery** даёт возможность находить инстансы по **логическому имени** сервиса и получать актуальный список сетевых адресов.

## Как сервисы находят друг друга

- **Реестр (Registry)** — центральный или распределённый компонент, в котором хранится «имя сервиса → список инстансов (host:port)».
- Сервисы при **старте** регистрируются в реестре (и периодически обновляют heartbeat); при остановке — снимаются с регистрации.
- **Клиент** (другой сервис или шлюз) запрашивает у реестра список инстансов по имени и выбирает один для вызова (сам — client-side discovery, или через прокси — server-side). В Spring Cloud типично **client-side**: OpenFeign/RestTemplate с LoadBalancer обращаются к реестру и выбирают инстанс сами.

# Eureka

## Что такое Eureka

**Eureka** — сервер **service discovery** из Netflix; в Spring Cloud доступен как **Spring Cloud Netflix Eureka**. Состоит из **Eureka Server** (реестр) и **Eureka Client** (в каждом сервисе). Сервер хранит регистрации инстансов; клиенты регистрируются при старте и периодически шлют heartbeat; при запросе «кто такой order-service?» сервер возвращает список инстансов. Клиентская библиотека Spring Cloud интегрирует Eureka с RestTemplate/WebClient и OpenFeign: вызов по имени сервиса (например, `http://order-service/api/orders`) превращается в запрос к одному из зарегистрированных инстансов.

## Как работает service registry

- Eureka Server хранит в памяти (с возможностью репликации между нодами) **регистрации**: application name, instance id, host, port, метаданные, статус (UP, DOWN), время последнего heartbeat.
- Клиенты отправляют **REGISTER** при старте, затем **HEARTBEAT** (по умолчанию каждые 30 сек). При отсутствии heartbeat в течение заданного времени (lease expiration) инстанс помечается как недоступный и исключается из ответов при запросах списка инстансов.
- Клиенты могут получать **полный реестр** (периодически) или запрашивать **конкретное приложение** по имени; обновления распространяются через периодический fetch или через механизм Eureka.

## Регистрация сервисов

- В микросервисе добавляют зависимость **eureka-client** (или spring-cloud-starter-netflix-eureka-client).
- В конфигурации указывают адрес Eureka Server: `eureka.client.service-url.defaultZone=http://eureka-server:8761/eureka/`.
- При старте приложения **EurekaClient** (или **DiscoveryClient**) регистрирует текущий инстанс: application name (из spring.application.name), host, port (из server.port или eureka.instance), instance id. Регистрация повторяется при heartbeat; при graceful shutdown можно отправить запрос на снятие с регистрации.

## Обнаружение сервисов

- Другой сервис (или шлюз) с Eureka Client запрашивает у Eureka список инстансов по **application name** (например, `order-service`). Eureka возвращает список с host:port.
- **Spring Cloud LoadBalancer** при вызове по имени (например, через RestTemplate с @LoadBalanced или OpenFeign с именем сервиса) обращается к DiscoveryClient, получает список инстансов, выбирает один (round-robin по умолчанию) и подставляет URL (например, `http://host:port/api/orders`). Таким образом, код вызывает `http://order-service/api/orders`, а реальный хост подставляется из реестра.

# API Gateway

## Зачем нужен API Gateway

- **Единая точка входа** для клиентов (веб, мобильные приложения, партнёры). Внутренняя топология (количество и имена сервисов) скрыта.
- **Маршрутизация** — запрос по пути или заголовкам направляется в нужный микросервис.
- **Сквозные функции** — аутентификация (JWT, OAuth2), авторизация, rate limiting, логирование, ограничение размера запроса на одном месте.
- **Агрегация** — при необходимости один запрос клиента может быть разбит на несколько вызовов к сервисам и ответ собран в один (BFF-сценарий).

## Какие задачи решает

- Скрытие внутренней структуры; один базовый URL для клиента.
- Централизованная аутентификация и проверка токенов до попадания в сервисы.
- Ограничение частоты запросов и защита от перегрузки.
- Упрощение клиентского кода и возможность менять внутренние сервисы без смены контракта для клиента.

# Spring Cloud Gateway

## Как работает Gateway

**Spring Cloud Gateway** — API Gateway на **Spring WebFlux** (реактивный стек). Принимает HTTP-запросы, сопоставляет их с **маршрутами** (Route), применяет **фильтры** (Gateway Filter) и отправляет запрос в целевой сервис (по URL или по имени из discovery). Ответ от сервиса может обрабатываться фильтрами и возвращается клиенту. Маршруты и фильтры задаются в конфигурации (YAML/properties) или в коде.

## Routing

- **Маршрут** задаётся: **id**, **uri** (целевой адрес или `lb://service-name` для LoadBalancer по имени сервиса), **predicates** (условия: path, method, header, host и т.д.), **filters** (модификация запроса/ответа).
- Пример: все запросы с path `/api/orders/**` направлять в `lb://order-service` (балансировка по инстансам order-service из discovery). Predicate по path: `Path=/api/orders/**`; uri: `lb://order-service`.
- Порядок маршрутов важен: первый подходящий predicate выбирает маршрут.

Пример конфигурации (YAML):

```yaml
spring:
  cloud:
    gateway:
      routes:
        - id: order-service
          uri: lb://order-service
          predicates:
            - Path=/api/orders/**
        - id: user-service
          uri: lb://user-service
          predicates:
            - Path=/api/users/**
```

## Filters

- **Gateway Filter** — логика до или после отправки запроса к целевому сервису: добавление/удаление заголовков, логирование, изменение пути, ограничение размера тела. Встроенные фильтры: AddRequestHeader, RemoveRequestHeader, RewritePath, Retry, RequestRateLimiter и др.
- **Global Filter** — применяется ко всем маршрутам; можно привязать логику к порядку (order). Типичное использование: извлечение JWT, добавление заголовка с user id для внутренних сервисов, логирование запроса/ответа.
- Фильтры выполняются в цепочке: pre-логика → запрос к сервису → post-логика → ответ клиенту.

## Security

- Аутентификация на Gateway: проверка JWT (например, через фильтр или интеграцию с Spring Security). При успехе можно добавить заголовки (X-User-Id, X-Roles) для внутренних сервисов.
- Spring Cloud Gateway совместим с **Spring Security**: можно включить OAuth2 Resource Server, проверку токенов и ограничение доступа по путям. Rate limiting — через фильтр RequestRateLimiter (часто с Redis).

# Configuration Management

## Зачем нужна централизованная конфигурация

- **Один источник правды** — настройки (порты БД, feature flags, URL внешних систем) хранятся в одном месте (например, Git или Vault), а не в десятках конфигов по сервисам.
- **Обновление без пересборки** — при поддержке refresh (Spring Cloud Config + Actuator refresh или Bus) можно обновить свойства без перезапуска приложения (в пределах возможностей приложения).
- **Окружения** — разные профили (dev, prod) и ветки/папки в репозитории конфигурации; один и тот же артефакт приложения разворачивается с разными наборами свойств.

## Проблемы локальных конфигураций

- Дублирование и рассинхрон: одно и то же свойство в многих репозиториях и инстансах.
- Секреты в репозитории приложения или в множестве файлов — сложнее ротация и контроль доступа.
- Изменение конфигурации требует пересборки или ручного изменения на каждой ноде.

# Spring Cloud Config

## Config Server

**Config Server** — отдельное Spring Boot-приложение с **spring-cloud-config-server**. Предоставляет REST API: по имени приложения (spring.application.name) и профилю (например, prod) отдаёт свойства (properties или YAML). Источники конфигурации: **Git** (типично), файловая система, Vault. Сервер читает конфиг из источника (например, репозиторий Git по ветке и пути) и отдаёт его клиентам. Поддерживает шифрование чувствительных значений (симметричное или асимметричное).

## Config Client

В каждом микросервисе добавляют **spring-cloud-starter-config** и настраивают адрес Config Server: `spring.config.import=optional:configserver:http://config-server:8888` (или bootstrap-свойства). При старте клиент запрашивает конфигурацию у сервера по своему `spring.application.name` и активным профилям; полученные свойства подмешиваются в Environment приложения. Таким образом, application.yml в приложении может содержать только минимальный набор (имя приложения, адрес Config Server), остальное — с сервера.

## Хранение конфигурации в Git

- В Config Server указывают URI репозитория Git, ветку (по умолчанию main), опционально путь к папке.
- Файлы в репозитории именуются: `{application}.yml`, `{application}-{profile}.yml` (например, `order-service.yml`, `order-service-prod.yml`). Сервер отдаёт свойства для запрошенного application и profile.
- Преимущества: версионирование конфигурации, код-ревью изменений, один репозиторий для всех сервисов или отдельный репозиторий конфигов. Секреты лучше хранить в Vault или шифровать значения в Git через Config Server.

# Inter-Service Communication

## Синхронное взаимодействие

Сервис A вызывает сервис B по **HTTP** и **ждёт ответа**. Реализуется через REST: клиент (RestTemplate, WebClient, OpenFeign) отправляет запрос на URL сервиса B, получает ответ (JSON, XML). Плюсы: простая модель, актуальные данные «здесь и сейчас». Минусы: связность по доступности (B должен быть доступен), риск каскадных сбоев без таймаутов и circuit breaker. В Spring Cloud синхронные вызовы обычно идут через **OpenFeign** или **RestTemplate** с **LoadBalancer** и адресом из **discovery**.

## REST communication

- Контракт: URL, метод, заголовки, тело запроса/ответа. Сериализация (JSON) через Jackson.
- В микросервисах URL целевого сервиса часто задаётся **логическим именем** (например, `http://order-service`); клиент (с LoadBalancer) подставляет реальный host:port из реестра. Таймауты, retry и circuit breaker настраиваются на клиенте (Resilience4j, настройки Feign).

# OpenFeign

## Что такое OpenFeign

**OpenFeign** — декларативный **HTTP-клиент**: интерфейс с аннотациями методов (URL, метод HTTP, заголовки, тело). В runtime генерируется реализация, которая выполняет HTTP-запросы. В Spring Cloud OpenFeign интегрируется с **Spring Cloud LoadBalancer** и **Service Discovery**: при указании имени сервиса в URL (например, `@FeignClient(name = "order-service")`) клиент запрашивает у discovery список инстансов, выбирает один (балансировка) и выполняет запрос. Поддерживаются кодирование/декодирование, обработка ошибок, логирование; можно добавить Resilience4j (circuit breaker, retry) через конфигурацию.

## Как упрощает REST-коммуникацию между сервисами

- Не нужно вручную создавать RestTemplate/WebClient и парсить ответы; достаточно описать интерфейс и внедрить его как бин.
- Один аннотированный интерфейс заменяет множество вызовов: типобезопасность, единое место для URL и параметров.
- Интеграция с discovery и балансировкой: вызов по имени сервиса без знания host:port.

Пример:

```java
@FeignClient(name = "order-service")
public interface OrderClient {
    @GetMapping("/api/orders/{id}")
    Order getById(@PathVariable("id") Long id);

    @PostMapping("/api/orders")
    Order create(@RequestBody OrderRequest request);
}

// В сервисе
@Service
public class PaymentService {
    private final OrderClient orderClient;
    // при вызове orderClient.getById(1L) запрос пойдёт на один из инстансов order-service
}
```

# Load Balancing

## Зачем нужен load balancing

При нескольких инстансах одного сервиса запросы нужно **распределять** между ними (round-robin, по загрузке, по зоне и т.д.), чтобы не перегружать один инстанс и использовать ресурсы равномерно. В микросервисах без отдельного hardware/software балансировщика перед каждым сервисом часто используют **клиентскую балансировку**: клиент получает список всех инстансов из discovery и сам выбирает, куда отправить запрос.

## Как Spring Cloud балансирует запросы

- **Spring Cloud LoadBalancer** — клиентский балансировщик. Реализация **ServiceInstanceListSupplier** возвращает список инстансов (из Eureka, Consul или другого discovery). **ReactorLoadBalancer** (или аналог) выбирает один инстанс по выбранной стратегии (по умолчанию round-robin). Интеграция с **RestTemplate** (@LoadBalanced), **WebClient** (LoadBalancerExchangeFilterFunction) и **OpenFeign** (при включённом loadbalancer) подставляет выбранный host:port вместо имени сервиса. Таким образом, один и тот же код «вызвать order-service» при каждом вызове может попадать на другой инстанс.

# Fault Tolerance

## Проблемы отказов в микросервисах

- Сетевые задержки и таймауты; инстанс может быть недоступен или перегружен.
- Если вызывающий сервис не ограничивает ожидание (таймаут) и не ограничивает число одновременных вызовов к «проблемному» сервису, потоки и соединения исчерпываются — **отказ распространяется** на вызывающую сторону (каскадный сбой).

## Каскадные ошибки

**Каскадный сбой** — когда отказ одного сервиса (или его сильная задержка) приводит к отказу зависимых сервисов: они забивают пулы потоков/соединений в ожидании ответа и перестают обслуживать запросы. Защита: **таймауты**, **ограничение параллельных вызовов** (bulkhead), **circuit breaker** (временно не вызывать сбойный сервис), **retry** с ограничениями и **fallback** (вернуть запасное значение или ошибку вместо бесконечного ожидания).

# Resilience4j

## Паттерны устойчивости

**Resilience4j** — библиотека для повышения отказоустойчивости: circuit breaker, retry, rate limiter, bulkhead, таймауты. Интегрируется с Spring Boot (стартер resilience4j-spring-boot2) и с OpenFeign. Конфигурация через properties или код.

## Circuit Breaker

При серии неудачных вызовов (или медленных) «размыкает цепь»: следующие вызовы **не отправляются** к целевому сервису, а сразу возвращают ошибку или fallback. Через заданное время состояние переходит в half-open: разрешается ограниченное число пробных вызовов; при успехе — closed (нормальная работа), при неудаче — снова open. Таким образом, сбойный сервис не нагружается и вызывающая сторона не блокируется.

## Retry

Повтор вызова при неудаче (исключение или определённый результат) с ограничением числа попыток и задержкой (фиксированной или exponential backoff). Уменьшает влияние временных сбоев; для неидемпотентных операций нужно учитывать риск дублирования.

## Rate Limiter

Ограничение частоты вызовов (например, не более N запросов в секунду к определённому ресурсу). Защита от перегрузки своего или внешнего сервиса.

## Bulkhead

Ограничение количества **параллельных вызовов** (или пула потоков) к определённому сервису или группе операций. Исчерпание пула не блокирует остальные вызовы в приложении — изоляция по «отсекам». Один перегруженный или сбойный сервис не забирает все ресурсы.

Пример конфигурации (application.yml) и использования с Feign:

```yaml
resilience4j:
  circuitbreaker:
    instances:
      orderService:
        registerHealthIndicator: true
        slidingWindowSize: 10
        failureRateThreshold: 50
        waitDurationInOpenState: 10s
  retry:
    instances:
      orderService:
        maxAttempts: 3
        waitDuration: 1s
```

# Distributed Tracing

## Проблемы отладки микросервисов

Один запрос пользователя может проходить через **несколько сервисов**. Без общего идентификатора невозможно связать логи и метрики разных сервисов в один «след» запроса. При ошибке или высокой задержке сложно понять, на каком сервисе и на каком этапе возникла проблема.

## Distributed tracing

**Распределённая трассировка** — механизм присвоения каждому запросу **уникального идентификатора** (trace id) и передачи его между сервисами (в HTTP-заголовках). Каждый сервис создаёт **span** (шаг обработки: входящий запрос, вызов другого сервиса, запрос к БД) и отправляет span в систему трассировки. Система собирает span'ы по trace id и строит **дерево вызовов**: какой сервис сколько времени занял, где произошла ошибка. Упрощает отладку и поиск узких мест.

## Zipkin и Jaeger

- **Zipkin** — система сбора и отображения трассировок. Сервер принимает span'ы (HTTP или Kafka), хранит их и предоставляет UI для просмотра по trace id. Spring Cloud Sleuth (устаревший) и **Micrometer Tracing** с bridge для Zipkin отправляют span'ы из приложения.
- **Jaeger** — аналог Zipkin; совместим с OpenTracing/OpenTelemetry. Концепция та же: агент/коллектор принимает span'ы, хранит, UI для запросов. В Spring приложениях трассировка обычно настраивается через **Micrometer Tracing** и экспорт в Zipkin или Jaeger.

# Monitoring

## Мониторинг микросервисов

- **Health** — живы ли инстансы (Actuator /health); Eureka использует health для снятия нездоровых инстансов с маршрутизации.
- **Метрики** — throughput, задержки, ошибки, использование ресурсов (CPU, память, пулы). Сбор с каждого инстанса и агрегация в централизованной системе.
- **Алерты** — при превышении порогов (ошибки, задержка, недоступность) уведомление команды. Типичная связка: метрики в **Prometheus**, правила алертинга, **Grafana** для дашбордов и визуализации.

## Prometheus и Grafana

- **Prometheus** — сбор метрик по pull-модели (опрос эндпоинтов приложений), хранение временных рядов, запросы PromQL. Spring Boot Actuator предоставляет эндпоинт /actuator/prometheus; Prometheus периодически забирает метрики.
- **Grafana** — дашборды по данным из Prometheus (и других источников): графики, таблицы, алерты. Удобно для обзора состояния сервисов, задержек и ошибок по инстансам и эндпоинтам.

# Messaging

## Event-driven архитектура

Вместо синхронного «вызвать сервис и ждать ответа» сервис **публикует событие** (сообщение) в очередь или топик. Другие сервисы **подписаны** на события и обрабатывают их асинхронно. Связность по времени снижается: издатель не знает потребителей; добавление нового потребителя не меняет издателя. Очередь буферизует нагрузку и может гарантировать доставку (в зависимости от брокера и настроек).

## Асинхронная коммуникация

- Обмен через **брокер сообщений** (RabbitMQ, Kafka) или через **событийную шину**. Формат сообщения — JSON, Avro и т.д.
- В Spring: **Spring Cloud Stream** или **Spring Kafka** / **Spring AMQP**. Публикация: вызов метода, который отправляет сообщение в канал/топик. Потребление: метод-слушатель вызывается при получении сообщения; при необходимости идемпотентность и обработка дублей.

# Kafka

## Зачем используется Kafka в микросервисах

- **Высокая пропускная способность** — подходит для потоков событий (логи, телеметрия, события домена).
- **Хранение лога** — сообщения хранятся в топике; новые потребители могут читать историю (в пределах retention).
- **Масштабирование** — партиционирование топика и consumer groups: несколько инстансов потребителя обрабатывают партиции параллельно.
- **Развязка** — производители и потребители не знают друг о друге; контракт — формат сообщения в топике.

## Преимущества event-driven архитектуры

- Слабая связность; возможность добавлять новые сервисы-потребители без изменения издателей.
- Устойчивость к временной недоступности потребителя (очередь буферизует).
- Масштабирование по потребителям; воспроизведение и повторная обработка при использовании Kafka.

В Spring Cloud сценарии часто реализуют через **Spring Cloud Stream** с биндером Kafka или напрямую через **Spring Kafka** (KafkaTemplate, @KafkaListener).

# Best Practices

## Централизованная конфигурация

- Использовать **Config Server** и хранить конфигурацию в Git (или Vault для секретов). Один артефакт приложения — разные окружения за счёт профилей и конфигурации с сервера.
- Не хранить секреты в открытом виде в репозитории; использовать шифрование Config Server или внешнее хранилище секретов.

## API Gateway

- Единая точка входа для внешних клиентов; маршрутизация по имени сервиса (lb://service-name) с discovery. Аутентификация и авторизация на шлюзе; внутренние сервисы получают обогащённые заголовки (user id, роли) при необходимости.
- Rate limiting и ограничение размера запроса для защиты от перегрузки и злоупотреблений.

## Observability

- **Трассировка** — включить распределённую трассировку (Micrometer Tracing, передача trace id в заголовках) и экспорт в Zipkin/Jaeger. Использовать один trace id в логах всех сервисов.
- **Метрики** — Actuator + Prometheus; дашборды в Grafana; алерты на ошибки, задержки и доступность.
- **Логи** — структурированные логи с trace id; централизованный сбор (ELK, Loki) для поиска по запросу.

## Resilience patterns

- **Таймауты** на все внешние вызовы (Feign, RestTemplate, клиенты БД).
- **Circuit breaker** для вызовов к критичным зависимостям; fallback или понятная ошибка при открытом контуре.
- **Retry** с ограничением и backoff для временных сбоев; учитывать идемпотентность операций.
- **Bulkhead** или ограничение параллелизма для изоляции «шумных» зависимостей.

# Краткий конспект для собеседования

## Service Discovery

- Сервисы регистрируются в **реестре** (Eureka, Consul) при старте; клиенты запрашивают список инстансов по **имени сервиса**. В Spring Cloud вызов по имени (например, OpenFeign с name = "order-service") через **LoadBalancer** подставляет реальный host:port из реестра. Решает проблему динамических адресов в облаке.

## API Gateway

- **Единая точка входа**: маршрутизация по пути к сервисам (в т.ч. lb://service-name), аутентификация, rate limiting, логирование. **Spring Cloud Gateway** — реактивный шлюз на WebFlux; маршруты (predicates, uri), фильтры (глобальные и на маршруте), интеграция с discovery и безопасностью.

## Config Server

- **Централизованная конфигурация**: Config Server отдаёт свойства по имени приложения и профилю; источник — Git (или файлы, Vault). Клиенты при старте запрашивают конфиг с сервера (spring.config.import или bootstrap); обновление без пересборки — через refresh/Bus при поддержке.

## OpenFeign

- **Декларативный REST-клиент**: интерфейс с @FeignClient(name = "service-name") и аннотированными методами (GetMapping, PostMapping и т.д.). Интеграция с LoadBalancer и discovery — вызов по имени сервиса. Упрощает вызовы между микросервисами; при необходимости добавляют Resilience4j (circuit breaker, retry).

## Resilience patterns

- **Circuit breaker** — при серии ошибок перестать вызывать сервис (open), затем пробовать (half-open). **Retry** — повтор с ограничением и backoff. **Rate limiter** — ограничение частоты. **Bulkhead** — ограничение параллельных вызовов. **Таймауты** — на всех внешних вызовах. Реализация в Spring Cloud — **Resilience4j**; интеграция с Feign и Spring Boot.
