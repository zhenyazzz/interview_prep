# Введение в System Design

## Что такое System Design

**System Design** — процесс определения **архитектуры системы**: компоненты, их взаимодействие, выбор технологий и компромиссов с учётом требований к нагрузке, доступности, согласованности данных и развитию в будущем. Включает масштабирование, отказоустойчивость, кэширование, очереди, репликацию и шардирование БД и другие темы.

## Почему это важно для backend-разработчиков

- **Собеседования** — на технических интервью часто дают задачу спроектировать систему (чат, лента, хранение файлов) и обсуждают trade-off.
- **Практика** — понимание, зачем нужны кэш, балансировщики, репликация и очереди, помогает принимать решения в реальных проектах и объяснять архитектуру.
- **Масштаб** — при росте нагрузки и числа сервисов важно знать, как не создавать узкие места и как обеспечивать доступность и согласованность.

## Какие проблемы решаются на уровне архитектуры

- **Нагрузка** — как распределять запросы и данные, чтобы система выдерживала рост трафика (масштабирование, кэш, шардирование).
- **Доступность** — как избежать единых точек отказа и быстро восстанавливаться (репликация, балансировка, health checks).
- **Согласованность** — как хранить и синхронизировать данные в распределённой среде (модели согласованности, репликация, очереди).
- **Наблюдаемость** — как понимать, что происходит в системе (логи, метрики, трассировка).

# Масштабируемость

## Что такое scalability

**Масштабируемость (Scalability)** — способность системы **увеличивать производительность** при росте нагрузки за счёт добавления ресурсов (узлов, CPU, памяти) без принципиального изменения архитектуры. Система масштабируема, если при удвоении нагрузки можно сохранить время отклика и пропускную способность, добавив ресурсы.

## Какие проблемы возникают при росте

- **Узкие места** — один компонент (БД, один сервер приложений, один диск) ограничивает пропускную способность всей системы.
- **Рост задержек** — при перегрузке очереди растут, время ответа увеличивается.
- **Отказы** — при превышении лимитов возможны таймауты, падения сервисов и каскадные сбои.
- **Данные** — объём данных растёт; одна БД может не справляться с записью и чтением.

Решения: горизонтальное масштабирование (больше инстансов), кэширование, репликация и шардирование БД, асинхронная обработка через очереди, CDN для статики.

# Vertical vs Horizontal Scaling

## Vertical scaling (scale up)

**Вертикальное масштабирование** — увеличение ресурсов **одного узла**: больше CPU, памяти, диска. Приложение остаётся на одном сервере, но сервер становится мощнее.

**Плюсы:** проще с точки зрения приложения (нет распределённости), меньше изменений в коде. **Минусы:** есть физический и финансовый предел; единая точка отказа; простой при обновлении железа.

## Horizontal scaling (scale out)

**Горизонтальное масштабирование** — добавление **новых узлов** (серверов приложений, реплик БД). Нагрузка распределяется между узлами; можно добавлять узлы по мере роста.

**Плюсы:** теоретически неограниченный рост, отказоустойчивость (при падении одного узла работают остальные). **Минусы:** нужны балансировка, общее состояние (сессии, кэш) или отказ от него, распределённые данные и согласованность.

## Таблица сравнения

| Критерий | Vertical (scale up) | Horizontal (scale out) |
|----------|---------------------|-------------------------|
| **Способ** | Больше CPU/RAM на одном сервере | Больше серверов |
| **Сложность** | Ниже (один узел) | Выше (балансировка, состояние, данные) |
| **Предел** | Ограничен возможностями железа | Ограничен в основном бюджетом и архитектурой |
| **Отказоустойчивость** | Один узел — единая точка отказа | Можно терять часть узлов |
| **Типичное использование** | На старте, быстрый рост мощности | Долгосрочное масштабирование, облако |

В облаке и для долгосрочного роста обычно выбирают **горизонтальное** масштабирование.

# High Availability

## Что такое высокая доступность

**High Availability (HA)** — свойство системы оставаться **доступной** большую часть времени (целевые показатели — 99.9%, 99.99% и т.д.). Достигается за счёт устранения единых точек отказа и быстрого переключения на резерв при сбое.

## Как её достигают

- **Резервирование** — несколько инстансов приложения и БД; при падении одного трафик идёт на другие.
- **Балансировка нагрузки** — распределение запросов по узлам; проверка здоровья узлов (health check) и исключение неработающих из пула.
- **Репликация БД** — мастер + реплики; при падении мастера одна из реплик становится новым мастером (failover).
- **Мониторинг и алертинг** — обнаружение сбоев и автоматическое или ручное переключение.
- **Graceful degradation** — при частичном отказе система отключает часть функций, но продолжает обслуживать критичные запросы.

```
  [Клиенты]
       │
       ▼
  [Load Balancer]  ─── health check
       │
   ┌───┴───┬───────┬───────┐
   ▼       ▼       ▼       ▼
 [App 1] [App 2] [App 3] [App 4]   ← несколько инстансов
   │       │       │       │
   └───────┴───┬───┴───────┘
               ▼
        [DB Master] ──► [Replica 1], [Replica 2]
```

# Load Balancing

## Что такое балансировка нагрузки

**Load Balancing** — распределение входящих запросов между **несколькими серверами** (пулом), чтобы ни один узел не был перегружен и трафик обрабатывался с приемлемой задержкой. Балансировщик принимает запрос и направляет его на один из доступных узлов по выбранному алгоритму.

## Зачем она нужна

- Равномерная загрузка узлов; использование всех инстансов.
- Отказоустойчивость: не отправлять трафик на недоступные или нездоровые узлы (по результатам health check).
- Масштабируемость: добавление новых узлов в пул без смены адреса для клиента (клиент обращается к балансировщику).

## Алгоритмы балансировки

| Алгоритм | Описание |
|----------|----------|
| **Round Robin** | Запросы по очереди на каждый узел по кругу. Просто; не учитывает текущую загрузку узлов. |
| **Least Connections** | Запрос отправляется на узел с наименьшим числом активных соединений. Подходит для долгих сессий (WebSocket, длинные запросы). |
| **IP Hash** | Узел выбирается по хешу IP клиента. Один и тот же клиент всегда попадает на один узел (sticky session через хеш). |

Дополнительно: weighted round robin (разный вес узлов), least response time (по средней задержке ответа).

## Примеры инструментов

- **NGINX** — reverse proxy и балансировщик; конфигурация upstream с перечислением серверов и выбором алгоритма (round_robin, least_conn, ip_hash). Health checks, SSL termination.
- **HAProxy** — специализированный балансировщик; гибкая настройка алгоритмов, health checks, ACL, статистика. Часто используется перед пулом приложений или БД.

# Caching

## Зачем нужен кэш

- **Снижение нагрузки** на источник данных (БД, внешний API): повторяющиеся запросы обслуживаются из быстрого хранилища (память, Redis), без обращения к БД.
- **Уменьшение задержки** — чтение из памяти или локального кэша быстрее, чем запрос по сети к БД.
- **Сглаживание пиков** — при всплеске трафика кэш принимает часть запросов и защищает нижние уровни.

## Как кэш уменьшает нагрузку

Типичный сценарий: запрос приходит → проверка кэша по ключу (например, id сущности) → при попадании (cache hit) ответ отдаётся из кэша; при промахе (cache miss) данные берутся из БД, сохраняются в кэш с TTL и отдаются клиенту. Доля cache hit определяет, насколько снижается нагрузка на БД. Кэш нужно инвалидировать или обновлять при изменении данных (запись в БД → удаление/обновление ключа в кэше).

## Типы кэша

| Тип | Описание |
|-----|----------|
| **Application cache** | Кэш в памяти процесса приложения (например, локальная Map или Caffeine). Быстро; не разделяется между инстансами; при нескольких репликах возможна рассинхронизация. |
| **Database cache** | Кэш на стороне БД (буферный пул, кэш запросов). Прозрачен для приложения; уменьшает чтения с диска. |
| **Distributed cache** | Отдельный сервис кэша (Redis, Memcached), общий для всех инстансов приложения. Единое кэш-хранилище при горизонтальном масштабировании. |

## Примеры технологий

- **Redis** — хранилище «ключ–значение» в памяти с персистентностью (опционально), структурами данных (строки, списки, множества, хеши), TTL, pub/sub. Часто используется как распределённый кэш и хранилище сессий.
- **Memcached** — простой распределённый кэш в памяти (строки по ключу), без персистентности. Низкая задержка, хорош для кэширования объектов по ключу.

# CDN

## Что такое Content Delivery Network

**CDN** — сеть **географически распределённых серверов** (edge-узлов), которые хранят копии статического контента (картинки, CSS, JS, видео) и отдают его пользователям с ближайшего узла. Запрос пользователя обрабатывается edge-сервером в его регионе, а не одним центральным сервером.

## Как CDN ускоряет доставку

- **Меньше задержка** — данные отдаются с ближайшего узла (меньше RTT).
- **Разгрузка origin** — запросы к статике не доходят до основного сервера; origin отдаёт контент только при промахе кэша CDN (cache miss) или при обновлении.
- **Масштабирование по географии** — один origin может обслуживать пользователей по всему миру за счёт кэша на edge.

Типичный поток: запрос к `cdn.example.com/image.jpg` → DNS возвращает IP ближайшего edge → edge отдаёт файл из своего кэша или запрашивает у origin, кэширует и отдаёт. TTL на edge задаётся заголовками (Cache-Control) или настройками CDN.

# Database Replication

## Что такое репликация БД

**Репликация** — копирование данных с одной БД (мастер/primary) на одну или несколько других (реплики/secondary) с целью распределения нагрузки на чтение, отказоустойчивости и географического распределения. Запись обычно идёт только на мастер; реплики получают изменения через лог репликации (WAL, binlog) и применяют их у себя.

## Зачем она используется

- **Чтение** — запросы на чтение можно направлять на реплики; мастер разгружается, пропускная способность по чтению растёт.
- **Отказоустойчивость** — при падении мастера одна из реплик может быть повышена до мастера (failover); данные не теряются при корректной настройке синхронной/полусинхронной репликации.
- **География** — реплики в разных регионах уменьшают задержку чтения для локальных пользователей.

## Master-slave и read replicas

- **Master-slave (primary-replica)** — одна запись (master), несколько реплик (slaves). Запись только на master; репликация асинхронная или синхронная. Чтение можно распределять по репликам.
- **Read replicas** — те же реплики, акцент на том, что они используются **только для чтения**. Приложение направляет SELECT на реплики, INSERT/UPDATE/DELETE — на master. Задержка репликации приводит к eventual consistency: сразу после записи на master чтение с реплики может ещё не показать новые данные.

```
  [Приложение]
       │
   write │ read
       │ │
       ▼ ▼
  [Master] ──── replication ───► [Replica 1]
       │                           [Replica 2]
       │                           [Replica 3]
```

# Database Sharding

## Что такое sharding

**Sharding (партиционирование по горизонтали)** — разбиение **набора данных** на части (шарды), каждая из которых хранится в отдельной БД (или группе реплик). Данные распределяются по шардам по ключу шардирования (user_id, region_id, hash(id) и т.д.). Запросы направляются в тот шард, где лежат нужные данные.

## Зачем делить базу данных

- **Объём** — одна БД не справляется с объёмом данных или с количеством записей/чтений; шарды распределяют и данные, и нагрузку.
- **Масштабирование записи** — при репликации запись всё равно идёт на один мастер; шардирование даёт несколько независимых мастеров и увеличивает пропускную способность записи.

## Какие проблемы решает и создаёт

**Решает:** рост объёма данных и нагрузки по записи/чтению; возможность масштабировать по шардам (добавлять новые шарды и перераспределять данные).

**Создаёт:** сложность запросов, затрагивающих несколько шардов (join между шардами, агрегации по всей выборке); необходимость выбора ключа шардирования и перебалансировки при добавлении шардов; операции по схеме (миграции) нужно выполнять на всех шардах.

# CAP Theorem

## Три свойства

| Свойство | Описание |
|----------|----------|
| **Consistency (согласованность)** | Каждый запрос на чтение получает последнюю записанную версию данных или ошибку. Все узлы видят одни и те же данные в один момент времени. |
| **Availability (доступность)** | Каждый запрос (на чтение или запись) получает ответ (успех или ошибка), система не «зависает» в ожидании. |
| **Partition Tolerance (устойчивость к разбиению)** | Система продолжает работать при **разделении сети** (partition): часть узлов не может общаться с другой частью. В реальных распределённых системах разбиение возможно всегда. |

## Почему невозможно обеспечить все три одновременно

При **разбиении сети** (P) два узла не могут обмениваться данными. Если узел A принял запись, а узел B из другой части сети не знает о ней:

- Чтобы сохранить **согласованность (C)** — узел B не должен отдавать устаревшие данные; он должен либо ждать связи с A (тогда ответ клиенту задерживается или невозможен), либо вернуть ошибку. В обоих случаях **доступность (A)** для части запросов теряется.
- Чтобы сохранить **доступность (A)** — узел B должен отвечать на запросы; тогда он может отдать старые данные, и **согласованность (C)** нарушается.

Вывод: при наличии разбиения (P) можно гарантировать либо **C**, либо **A**, но не оба сразу. На практике P не отбрасывают (сети ненадёжны), поэтому выбирают между CP (например, строгая согласованность, отказ в ответе при разбиении) и AP (доступность, допускается eventual consistency). Упрощённо: **CAP** = при partition выбираем C или A.

# Consistency Models

## Strong consistency

**Строгая согласованность** — после успешной записи любое последующее чтение (с любого узла) возвращает эту запись. Нет «окна» устаревания. Достигается за счёт блокировок, консенсусных протоколов (Paxos, Raft) или синхронной репликации перед подтверждением записи клиенту. Цена — задержка и снижение доступности при сбоях и разбиениях.

## Eventual consistency

**Согласованность в конечном счёте** — после записи система **со временем** переходит в согласованное состояние: когда распространение обновлений завершится, все узлы увидят одни и те же данные. До этого момента чтение может вернуть старую версию. Используется для повышения доступности и снижения задержки (асинхронная репликация, кэш, распределённые хранилища с репликами). Подходит там, где допустима временная рассинхронизация.

---

# Eventual Consistency

## Как работает eventual consistency

- Запись применяется на одном узле (или на нескольких с консенсусом) и подтверждается клиенту.
- Обновления **распространяются асинхронно** на другие узлы (реплики, кэш, очереди). Клиент не ждёт завершения распространения.
- Чтение с любого узла может вернуть старую версию, пока обновление не дошло до этого узла. После распространения все узлы имеют одну и ту же версию (в отсутствие новых записей).
- Конфликты (одновременная запись в один ключ с разных узлов) разрешаются политикой (last-write-wins, версии, CRDT и т.д.).

## Где используется

- Реплики БД с асинхронной репликацией (read replicas).
- Распределённый кэш (обновление и инвалидация с задержкой).
- Распределённые хранилища (Dynamo-style, Cassandra): запись на N узлов, чтение с R узлов; со временем реплики синхронизируются.
- Событийная синхронизация между сервисами (события доставляются с задержкой).

# Message Queues

## Зачем используются message brokers

- **Развязка** — производитель и потребитель не вызывают друг друга напрямую; сообщение кладётся в очередь и обрабатывается асинхронно. Производитель не блокируется на обработке.
- **Буферизация** — пики нагрузки сглаживаются очередью; потребители обрабатывают сообщения со своей скоростью.
- **Надёжность** — сообщения сохраняются в брокере; при падении потребителя сообщения не теряются и обрабатываются после восстановления.
- **Масштабирование** — можно добавлять потребителей и обрабатывать очередь параллельно.

## Какие проблемы решают

- Перегрузка синхронного вызова (долгая обработка на стороне сервиса блокирует вызывающего).
- Временная недоступность потребителя (очередь накапливает сообщения).
- Распределение нагрузки между несколькими воркерами (конкурирующие потребители).

## Примеры: Kafka и RabbitMQ

| Система | Модель | Особенности |
|---------|--------|-------------|
| **Kafka** | Лог сообщений (topic, partition), потребители читают с offset. | Высокая пропускная способность, хранение сообщений, повторное чтение; подходит для потоков событий и event sourcing. |
| **RabbitMQ** | Очереди и обменники (exchange → queue). | Гибкая маршрутизация (routing key, fanout, topic); классическая модель «очередь задач»; сообщение удаляется после доставки потребителю (при ack). |

# Event-driven Architecture

## Что такое event-driven архитектура

**Event-driven архитектура** — подход, при котором компоненты системы взаимодействуют через **события**: один компонент публикует событие (что произошло); другие подписаны на типы событий и реагируют на них асинхронно. Взаимодействие идёт через брокер сообщений (Kafka, RabbitMQ) или событийную шину. Прямых синхронных вызовов между издателем и подписчиками нет.

## Преимущества асинхронной коммуникации

- **Слабая связность** — издатель не знает потребителей; добавление нового подписчика не меняет код издателя.
- **Масштабирование** — потребители масштабируются независимо; очередь буферизует нагрузку.
- **Отказоустойчивость** — при падении потребителя сообщения остаются в очереди; повторная обработка после восстановления.
- **Гибкость** — один поток событий может обслуживать несколько сценариев (аналитика, уведомления, обновление кэша и т.д.).

# Rate Limiting

## Что такое rate limiting

**Rate limiting** — ограничение **количества запросов** от клиента (или по ключу: IP, user id, API key) в единицу времени. При превышении лимита запрос отклоняется (429 Too Many Requests или аналогичный ответ). Цель — защита от перегрузки, злоупотреблений и DDoS; справедливое распределение ресурсов между клиентами.

## Зачем ограничивать запросы

- Защита сервиса от одного «шумного» клиента или бота.
- Обеспечение квот (тарифные планы API).
- Снижение риска brute-force (ограничение попыток входа, запросов на восстановление пароля).

## Алгоритмы

| Алгоритм | Описание |
|----------|----------|
| **Token Bucket** | Ведро с токенами пополняется с заданной скоростью; каждый запрос забирает один токен. Если токенов нет — отказ. Допускает кратковременные всплески (накопленные токены). |
| **Leaky Bucket** | Запросы поступают в «ведро»; обрабатываются с фиксированной скоростью (утекают как из дырявого ведра). Сглаживает всплески. |
| **Fixed Window** | Счётчик запросов за фиксированный интервал (например, минута). При превышении лимита — отказ. В начале нового окна счётчик сбрасывается; возможен всплеск на границе окон. |

**Sliding Window** — лимит за скользящий интервал (например, последние 60 секунд); сглаживает граничный эффект fixed window.

# Idempotency

## Что такое идемпотентность

**Идемпотентность** — свойство операции: **повторное выполнение с теми же входными данными даёт тот же результат**, что и однократное, и не создаёт дополнительных побочных эффектов. Например, повторная отправка одного и того же запроса «списать 100 руб.» не должна списывать 200 руб.

## Почему это важно в распределённых системах

- **Повторы запросов** — клиент может отправить запрос повторно при таймауте или сбое; сеть может дублировать пакеты. Без идемпотентности повтор приведёт к двойному списанию, двойному созданию заказа и т.д.
- **Повторы при доставке сообщений** — потребитель очереди может обработать одно и то же сообщение дважды (at-least-once). Обработка должна быть идемпотентной: проверка «уже обработано» по ключу (idempotency key) или по бизнес-идентификатору.

Реализация: клиент передаёт **idempotency key** (уникальный идентификатор запроса); сервер сохраняет результат по этому ключу и при повторном запросе с тем же ключом возвращает сохранённый результат без повторного выполнения операции.

# Distributed Locks

## Зачем нужны распределённые блокировки

В распределённой системе несколько инстансов приложения или сервисов могут одновременно пытаться выполнить **критическую операцию** (например, обработать один и тот же элемент очереди, изменить один ресурс). Чтобы только один из них выполнял операцию в данный момент, нужна **блокировка**, общая для всех узлов — то есть распределённый lock. Захват блокировки даёт право выполнить операцию; освобождение (или истечение TTL) даёт право другому узлу.

## Как их реализуют

- **Хранилище с атомарной операцией** — например, Redis: SET key value NX PX ttl (установить ключ только если его нет, с TTL). Узел, успешно установивший ключ, «держит» блокировку; по завершении работы удаляет ключ (release). TTL страхует от зависания при падении узла (lock автоматически освободится).
- **Консенсус и лидерство** — ZooKeeper, etcd: эфемерные узлы (znode), выбор лидера; блокировка через создание узла или последовательности узлов. Подходит для координации и выбора единственного исполнителя.

Важно: корректная обработка TTL и продления блокировки, чтобы не освободить lock раньше времени и не выполнить операцию дважды.

## Redis и ZooKeeper

- **Redis** — простые блокировки через SET NX PX; низкая задержка; подходит для коротких блокировок. Нет гарантий при падении мастера (данные могут потеряться до репликации); для критичных сценариев используют Redlock (несколько инстансов Redis) или более надёжные системы.
- **ZooKeeper** — согласованность (CP); эфемерные узлы исчезают при отключении клиента; последовательности узлов для fair locks. Используется в Kafka (координация брокеров), Hadoop и др. для координации и выбора лидера.

# Monitoring

## Зачем нужен мониторинг системы

- **Обнаружение сбоев** — падение сервиса, рост ошибок, недоступность зависимостей; алерты позволяют реагировать до эскалации.
- **Понимание нагрузки** — пропускная способность, задержки, использование ресурсов (CPU, память, диск, сеть); выявление узких мест.
- **Анализ трендов** — рост трафика, деградация производительности; планирование масштабирования и оптимизации.
- **SLA** — контроль выполнения соглашений по доступности и задержкам.

## Prometheus и Grafana

- **Prometheus** — сбор **метрик** по модели pull (опрос эндпоинтов приложений), хранение временных рядов, язык запросов PromQL, алертинг (Alertmanager). Приложения экспортируют метрики (например, /metrics в формате Prometheus); Prometheus периодически забирает их. Подходит для метрик доступности, задержек, счётчиков ошибок.
- **Grafana** — **дашборды** и визуализация: графики по данным из Prometheus (и других источников), алерты, аннотации. Удобно для обзора состояния сервисов и расследования инцидентов.

# Observability

## Три компонента наблюдаемости

**Observability** — возможность понимать внутреннее состояние системы по её **внешним выходам** (логи, метрики, трассировки). Три столпа:

| Компонент | Описание |
|-----------|----------|
| **Logs** | Текстовые записи о событиях: ошибки, запросы, ключевые шаги. Должны быть структурированы (JSON) и содержать общий идентификатор запроса (trace id, correlation id), чтобы связать записи одного запроса в разных сервисах. Сбор в централизованное хранилище (ELK, Loki). |
| **Metrics** | Числовые показатели во времени: количество запросов, задержки, ошибки, использование CPU/памяти. Агрегируются и хранятся в системе метрик (Prometheus). Используются для дашбордов и алертов. |
| **Tracing** | Распределённая трассировка: один запрос получает trace id; каждый сервис создаёт span (шаг обработки) и передаёт trace id дальше. Сбор span'ов (Zipkin, Jaeger) позволяет построить дерево вызовов: какой сервис сколько времени занял, где произошла ошибка. |

Вместе логи, метрики и трассировка дают возможность локализовать сбои и узкие места в распределённой системе.

# Краткий конспект

## Быстрое повторение

- **Scalability** — способность системы расти с нагрузкой за счёт добавления ресурсов. Вертикальное (больше ресурсов на узел) vs горизонтальное (больше узлов); для долгосрочного роста — горизонтальное.
- **Caching** — кэш снижает нагрузку на БД и задержку; типы: in-memory в приложении, кэш БД, распределённый кэш (Redis, Memcached). Важны инвалидация и TTL.
- **Load balancing** — распределение запросов по узлам; алгоритмы: Round Robin, Least Connections, IP Hash. Инструменты: NGINX, HAProxy. Нужен для горизонтального масштабирования и отказоустойчивости.
- **CAP theorem** — при разбиении сети (P) можно гарантировать либо согласованность (C), либо доступность (A). На практике выбирают CP или AP; строгая согласованность и полная доступность при partition одновременно недостижимы.
- **Sharding** — разбиение данных на шарды (отдельные БД); распределение по ключу шардирования. Решает рост объёма и нагрузки; усложняет запросы по нескольким шардам и миграции.
- **Replication** — копирование данных мастер → реплики; read replicas разгружают мастер по чтению и дают отказоустойчивость. Репликация часто асинхронная → eventual consistency при чтении с реплик.
- **High Availability** — достижение через резервирование (несколько инстансов), балансировку, репликацию БД, мониторинг и failover.
- **Message queues** — развязка и буферизация; Kafka (поток событий, повторное чтение), RabbitMQ (очереди, гибкая маршрутизация). Event-driven архитектура строится на событиях и очередях.
- **Rate limiting** — ограничение числа запросов по клиенту/ключу; алгоритмы: Token Bucket, Leaky Bucket, Fixed/Sliding Window. Защита от перегрузки и злоупотреблений.
- **Idempotency** — повторное выполнение операции с теми же данными не даёт дополнительного эффекта; важно при повторах запросов и сообщений; реализуется через idempotency key.
- **Distributed locks** — общая блокировка между узлами (Redis SET NX, ZooKeeper); для критичных операций с единственным исполнителем. Учитывать TTL и корректное освобождение.
- **Monitoring** — метрики (Prometheus), дашборды и алерты (Grafana). **Observability** — логи (с trace id), метрики, трассировка (Zipkin, Jaeger) для отладки распределённых систем.
