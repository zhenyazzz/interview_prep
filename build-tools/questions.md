# Вопросы и ответы: Maven и Gradle (собеседование Java Backend)

Вопросы и краткие ответы по Maven и Gradle для подготовки к техническим собеседованиям.

---

## 1. Вопросы по Maven

### Что такое Maven?

**Maven** — инструмент автоматизации сборки и управления проектами для Java. Обеспечивает единый жизненный цикл сборки, управление зависимостями через POM и репозитории, расширяемость через плагины. Основан на принципе **convention over configuration** (соглашения вместо настройки).

### Какую проблему решает Maven?

- **Воспроизводимость сборки** — одна команда (`mvn install`) вместо ручного вызова `javac` и копирования JAR.
- **Управление зависимостями** — автоматическое скачивание библиотек по координатам, разрешение транзитивных зависимостей.
- **Единая структура проекта** — стандартная раскладка каталогов и фаз, понятная всей команде.
- **Расширяемость** — плагины для компиляции, тестов, упаковки, деплоя, отчётов.

### Что такое pom.xml?

**pom.xml** (Project Object Model) — XML-файл в корне проекта, описывающий проект для Maven:

- координаты (groupId, artifactId, version);
- зависимости и их scope;
- плагины и настройки сборки;
- репозитории, профили, модули.

Maven использует POM для сборки и разрешения зависимостей.

### Что такое координаты Maven (coordinates)?

**Координаты** однозначно идентифицируют артефакт в репозитории:

| Координата   | Назначение                          | Пример              |
|--------------|-------------------------------------|---------------------|
| **groupId**  | Организация/проект (обратный домен) | `com.example`       |
| **artifactId** | Имя артефакта                     | `user-service`      |
| **version**  | Версия                              | `1.0.0`, `2.1-SNAPSHOT` |
| **packaging**| Тип упаковки (по умолчанию `jar`)   | `jar`, `war`, `pom` |

Формат: `groupId:artifactId:packaging:version`.

### Что такое артефакт в Maven?

**Артефакт** — результат сборки (JAR, WAR, EAR) или зависимость (библиотека), идентифицируемая координатами. Хранится в репозиториях в виде файлов вида `artifactId-version.packaging` (например, `spring-core-6.1.0.jar`).

### Что такое жизненный цикл сборки Maven?

**Жизненный цикл** — последовательность фаз, которые Maven выполняет при сборке. Три цикла:

- **clean** — очистка (удаление `target/`);
- **default** — компиляция, тесты, упаковка, установка, деплой;
- **site** — генерация документации/отчётов.

При вызове фазы выполняются она и все предыдущие фазы того же цикла.

### Какие основные фазы жизненного цикла Maven?

| Фаза       | Действие                                      |
|------------|------------------------------------------------|
| **validate** | Проверка корректности проекта                 |
| **compile** | Компиляция исходного кода                    |
| **test**    | Запуск юнит-тестов (Surefire)                 |
| **package** | Упаковка в JAR/WAR и т.д.                     |
| **verify**  | Проверки артефакта (например, интеграционные тесты) |
| **install** | Установка артефакта в локальный репозиторий   |
| **deploy**  | Публикация в удалённый репозиторий            |

Порядок: validate → compile → test → package → verify → install → deploy.

### Что такое плагины Maven?

**Плагины** — расширения, которые выполняют реальные действия (компиляция, тесты, упаковка). Содержат **goals** (цели), привязанные к фазам жизненного цикла. Примеры: `maven-compiler-plugin`, `maven-surefire-plugin`, `maven-jar-plugin`, `spring-boot-maven-plugin`. Без плагинов фазы не делали бы ничего.

### Что такое репозитории Maven?

**Репозиторий** — хранилище артефактов:

- **Локальный** — каталог на диске (обычно `~/.m2/repository`), кэш всех загруженных артефактов.
- **Удалённый** — сервер (Nexus, Artifactory, Maven Central); Maven скачивает артефакты при необходимости.

При разрешении зависимости Maven сначала ищет в локальном репозитории, затем в удалённых.

### Что такое Maven Central?

**Maven Central** — центральный публичный репозиторий (https://repo.maven.apache.org/maven2/). Содержит огромное количество открытых Java-библиотек. По умолчанию Maven подключается к нему для разрешения зависимостей. Публикация в Central требует соблюдения правил (подпись, метаданные).

### Что такое локальный репозиторий?

**Локальный репозиторий** — каталог на машине разработчика (по умолчанию `~/.m2/repository`). В него скачиваются артефакты из удалённых репозиториев и устанавливаются артефакты при `mvn install`. Один раз скачанная зависимость переиспользуется во всех проектах.

### Что такое scope зависимостей в Maven?

**Scope** определяет, на каких этапах зависимость доступна и попадает ли в артефакт:

| Scope       | Компиляция | Тесты | Runtime | В артефакт | Пример использования        |
|-------------|------------|-------|---------|------------|-----------------------------|
| **compile** | да         | да    | да      | да         | Основные библиотеки (по умолчанию) |
| **provided**| да         | да    | нет     | нет        | Servlet API, Lombok         |
| **runtime** | нет        | да    | да      | да         | JDBC-драйверы               |
| **test**    | нет        | да    | нет     | нет        | JUnit, Mockito              |

### Что такое транзитивные зависимости?

**Транзитивные зависимости** — зависимости зависимостей. Если проект зависит от библиотеки A, а A зависит от B, то B подтягивается автоматически. Дерево можно посмотреть: `mvn dependency:tree`. При конфликте версий Maven выбирает одну по правилам (nearest definition, first declaration).

### Что такое dependency management?

**Dependency management** — блок `<dependencyManagement>` в POM, где объявляются **версии** (и при необходимости scope) зависимостей без автоматического добавления их в проект. Модули или дочерние POM ссылаются на артефакт **без указания версии**. Используется в parent POM и с BOM для централизованного управления версиями.

### Что такое мультимодульный Maven-проект?

Проект, разбитый на несколько **модулей**, у каждого свой `pom.xml`. Корневой POM перечисляет модули в `<modules>` и может быть **parent** и/или **aggregator**:

- **Parent POM** — общие зависимости в `<dependencyManagement>`, плагины, свойства; модули подключают его через `<parent>`.
- **Aggregator** — сборка всех модулей одной командой из корня (`mvn clean install`).

### Что такое профили Maven?

**Профили** — наборы настроек (свойства, зависимости, плагины, репозитории), активируемые по условию или вручную (`-P prod`). Позволяют иметь разные конфигурации для dev, test, prod без дублирования проектов. Активация: по свойству, по ОС, по наличию файла, по JDK.

### Что происходит при выполнении `mvn clean install`?

1. **clean** — фаза цикла clean: удаляется каталог `target/`.
2. **install** — фаза цикла default: выполняются validate → compile → test → package → verify → **install** (артефакт копируется в локальный репозиторий `~/.m2/repository`).

Деплой в удалённый репозиторий при этом **не** выполняется (для этого нужна фаза `deploy`).

## 2. Вопросы по Gradle

### Что такое Gradle?

**Gradle** — система автоматизации сборки с открытым исходным кодом. Использует DSL (Groovy или Kotlin) вместо XML, модель на основе **задач (tasks)** и плагинов. Поддерживает инкрементальную сборку, кэширование и daemon. Широко используется в Java, Kotlin, Android.

### Какую проблему решает Gradle?

- **Гибкость** — произвольные сценарии сборки без жёсткого жизненного цикла как в Maven.
- **Скорость** — daemon, инкрементальная сборка, build cache, параллельное выполнение.
- **Удобный конфиг** — короткий DSL вместо многословного XML.
- **Один инструмент** — Java, Kotlin, Android, нативные проекты через плагины.

### Что такое build-скрипт Gradle?

**Build script** — файл `build.gradle` (Groovy) или `build.gradle.kts` (Kotlin) в корне проекта (и в каждом модуле). Описывает плагины, репозитории, зависимости, задачи и другие настройки проекта. Gradle выполняет скрипт на фазе конфигурации и строит граф задач.

### В чём разница между Groovy DSL и Kotlin DSL?

| Аспект       | Groovy DSL (build.gradle)   | Kotlin DSL (build.gradle.kts) |
|--------------|-----------------------------|--------------------------------|
| Файл         | build.gradle                | build.gradle.kts               |
| Типизация    | Динамическая                | Статическая                    |
| IDE          | Хорошая поддержка           | Отличная (рефакторинг, навигация) |
| Строки       | `'...'` или `"..."`        | `"..."`                        |
| Присваивание | `mainClass = '...'`         | `mainClass.set("...")` и т.п.  |

Оба делают одно и то же; Kotlin DSL даёт лучшую типобезопасность и автодополнение.

### Что такое задачи (tasks) в Gradle?

**Task** — атомарная единица работы: компиляция, копирование файлов, запуск тестов и т.д. У задачи есть имя, действия (`doFirst`/`doLast`) и зависимости от других задач. Gradle строит **граф задач** и выполняет их в правильном порядке. Задачи с объявленными inputs/outputs могут быть пропущены как UP-TO-DATE при инкрементальной сборке.

### Что такое плагины в Gradle?

**Плагины** — расширения, добавляющие в проект задачи, конфигурации зависимостей и конвенции (например, структуру каталогов). Примеры: `java`, `application`, `org.springframework.boot`. Подключаются через `plugins { id 'java' }` или `apply plugin: 'java'`. Упрощают build-скрипт и стандартизируют сборку.

### Что такое Gradle Wrapper?

**Gradle Wrapper** — скрипты `gradlew` (Unix) и `gradlew.bat` (Windows) плюс каталог `gradle/wrapper` с JAR и `gradle-wrapper.properties`. При первом запуске скачивается версия Gradle, указанная в свойствах. Гарантирует **одну и ту же версию Gradle** у всех разработчиков и в CI без установки Gradle в систему. Использование: `./gradlew build` вместо `gradle build`.

### Что такое Gradle Daemon?

**Gradle Daemon** — долгоживущий процесс JVM, в котором выполняются сборки. Следующий запуск `gradlew` переиспользует тот же процесс. Снижает время старта JVM и ускоряет повторные сборки за счёт кэширования загруженных классов и скриптов. По умолчанию включён (`org.gradle.daemon=true` в gradle.properties).

### Что такое конфигурации зависимостей в Gradle?

**Конфигурация** определяет, когда зависимость нужна (compile/test/runtime) и передаётся ли она потребителям проекта. Основные:

| Конфигурация        | Компиляция | Тесты | Runtime | Транзитивно | Аналог Maven   |
|---------------------|------------|-------|---------|-------------|----------------|
| **implementation** | да         | да    | да      | нет         | compile (скрыт)|
| **api**             | да         | да    | да      | да          | compile        |
| **compileOnly**     | да         | да    | нет     | нет         | provided       |
| **runtimeOnly**     | нет        | да    | да      | нет         | runtime        |
| **testImplementation** | нет    | да    | нет     | нет         | test           |

### Что такое инкрементальная сборка в Gradle?

Задачи объявляют **inputs** и **outputs**. Если они не изменились с прошлого запуска, задача помечается как **UP-TO-DATE** и не выполняется. Это ускоряет повторные сборки без полной перекомпиляции и перезапуска тестов. Плагины Java, Kotlin и др. задают входы/выходы для компиляции и упаковки.

### Что такое граф задач в Gradle?

**Граф задач (task graph)** — ориентированный ациклический граф (DAG) зависимостей между задачами. Зависимости задаются через `dependsOn`, `mustRunAfter`, `shouldRunAfter`. Gradle вычисляет порядок выполнения и выполняет каждую задачу один раз. При выполнении `gradle build` сначала выполняются все задачи, от которых зависит `build`.

### Какой командой собирают Gradle-проект?

Основная команда — **сборка** (компиляция + тесты + упаковка):

```bash
./gradlew build
```

или с глобально установленным Gradle:

```bash
gradle build
```

Рекомендуется использовать **gradlew**, чтобы версия Gradle была зафиксирована в проекте.

## 3. Maven vs Gradle: вопросы на собеседовании

### В чём разница между Maven и Gradle?

- **Конфигурация**: Maven — XML (pom.xml), Gradle — Groovy/Kotlin DSL (build.gradle / build.gradle.kts).
- **Модель**: Maven — фиксированные фазы жизненного цикла; Gradle — граф задач, гибкий порядок.
- **Скорость**: Gradle обычно быстрее за счёт daemon, кэша и инкрементальной сборки.
- **Гибкость**: В Gradle проще писать кастомные сценарии и задачи; в Maven сильнее опора на конвенции и плагины.
- **Зависимости**: оба поддерживают репозитории Maven и управление зависимостями; в Gradle — конфигурации (implementation, api и т.д.), в Maven — scope и dependencyManagement.

### Почему Gradle считается быстрее?

- **Daemon** — переиспользование JVM между сборками, меньше времени на старт.
- **Инкрементальная сборка** — пропуск задач с неизменными inputs/outputs (UP-TO-DATE).
- **Build cache** — переиспользование результатов задач (в т.ч. между проектами и машинами).
- **Параллельное выполнение** — независимые задачи и подпроекты выполняются параллельно.

### Почему Maven проще изучать?

- Один формат конфигурации — XML, знакомый многим.
- Жёсткий жизненный цикл — предсказуемая последовательность фаз.
- Много документации и примеров, давняя история в enterprise.
- Не нужно знать Groovy/Kotlin для базовой настройки.

### Какие стили конфигурации используют Maven и Gradle?

- **Maven** — декларативный XML: теги `<dependencies>`, `<build>`, `<plugins>`. Конвенция над конфигурацией.
- **Gradle** — императивный/декларативный DSL: вызовы методов, блоки, при необходимости произвольный код (Groovy/Kotlin).

### В чём разница между pom.xml и build.gradle?

| Аспект    | pom.xml              | build.gradle / build.gradle.kts   |
|----------|----------------------|-----------------------------------|
| Формат   | XML                  | Groovy или Kotlin DSL             |
| Зависимости | `<dependencies>` с `<dependency>` | `dependencies { implementation(...) }` |
| Плагины  | `<build><plugins>`   | `plugins { id '...' }`            |
| Задачи/фазы | Привязка goals к фазам | Задачи (tasks) и зависимости между ними |
| Расширяемость | Через плагины и профили | Плагины + произвольный код в скрипте |

### Сравнительная таблица: Maven и Gradle

| Критерий       | Maven                    | Gradle                          |
|----------------|--------------------------|---------------------------------|
| **Стиль конфигурации** | XML (pom.xml)        | Groovy/Kotlin DSL (build.gradle) |
| **Гибкость**   | Ограничена фазами и плагинами | Высокая, кастомные задачи и скрипты |
| **Производительность** | Средняя, полная пересборка чаще | Выше: daemon, кэш, инкрементальность |
| **Кривая обучения** | Ниже, предсказуемый цикл | Выше, нужно понимать задачи и DSL |
| **Экосистема** | Очень большая (Central, плагины) | Большая, те же репозитории Maven |
| **Типичное использование** | Enterprise, Spring, стандартизированные проекты | Android, сложные сборки, большие монорепо |

## 4. Когда использовать Maven, когда Gradle

### Когда предпочтительнее Maven

- **Корпоративные проекты** — устоявшиеся стандарты, много готовых Maven-интеграций (Nexus, CI, корпоративные родительские POM).
- **Стандартизированные сборки** — один и тот же жизненный цикл и конвенции во всех проектах.
- **Команды, предпочитающие convention over configuration** — минимум кастомизации, всё «как у всех».
- **Проекты с множеством Maven-ориентированных инструментов** — артефакты, деплой, отчёты уже завязаны на Maven.

### Когда предпочтительнее Gradle

- **Сложные сборки** — нестандартные шаги, условная логика, несколько артефактов из одного проекта.
- **Сильно кастомизируемая логика сборки** — скрипты на Groovy/Kotlin, тонкая настройка задач.
- **Android** — Gradle является стандартом сборки для Android.
- **Крупные сборки, где важна скорость** — мультимодульные проекты, частые инкрементальные сборки, CI с кэшем.

## 5. Использование в реальных проектах

| Область              | Maven | Gradle | Комментарий |
|----------------------|-------|--------|-------------|
| **Spring Boot**      | Да    | Да     | Оба официально поддерживаются; Spring Initializr предлагает оба. |
| **Микросервисы**     | Да    | Да     | Часто Maven в legacy, Gradle в новых проектах; оба подходят. |
| **Android**          | Редко | Да     | Gradle — стандарт сборки Android (AGP). |
| **Корпоративные системы** | Часто | Растёт | Maven исторически доминирует; Gradle внедряют ради скорости и гибкости. |
| **Библиотеки / OSS** | Часто | Да     | Maven Central и оба инструмента поддерживают публикацию. |

Кратко:

- **Spring Boot** — и Maven (pom.xml, spring-boot-maven-plugin), и Gradle (spring-boot-gradle-plugin) в документации и стартерах.
- **Микросервисы** — выбор зависит от команды и инфраструктуры; оба интегрируются с Docker, Kubernetes, CI/CD.
- **Android** — практически только Gradle (Android Gradle Plugin).
- **Enterprise** — Maven по-прежнему очень распространён; Gradle выбирают при необходимости производительности и кастомизации.

## 6. Практические команды

### Maven

| Команда | Действие |
|---------|----------|
| `mvn clean install` | Удаляет `target/`, выполняет сборку до фазы install (артефакт в локальный репозиторий). |
| `mvn package` | Сборка до фазы package (JAR/WAR в `target/`), без install и deploy. |
| `mvn dependency:tree` | Выводит дерево прямых и транзитивных зависимостей. |
| `mvn clean package -DskipTests` | Очистка и упаковка без запуска тестов. |
| `mvn compile` | Только компиляция исходного кода. |
| `mvn test` | Компиляция и запуск тестов. |

```bash
mvn clean install
mvn package
mvn dependency:tree
```

### Gradle

| Команда | Действие |
|---------|----------|
| `./gradlew build` | Полная сборка: компиляция, тесты, упаковка (jar/bootJar и т.д.). |
| `./gradlew clean` | Удаление каталога `build/`. |
| `./gradlew tasks` | Список доступных задач (с описаниями). |
| `./gradlew dependencies` | Дерево зависимостей по конфигурациям. |
| `./gradlew clean build` | Очистка и полная сборка. |
| `./gradlew bootRun` | Запуск Spring Boot приложения (при подключённом Spring Boot plugin). |

```bash
./gradlew build
./gradlew clean
./gradlew tasks
./gradlew dependencies
```

На Windows: `gradlew.bat` вместо `./gradlew`.

## 7. Шпаргалка перед собеседованием

### Ключевые концепции Maven

- **POM** — pom.xml, описание проекта и зависимостей.
- **Координаты** — groupId, artifactId, version (и packaging).
- **Жизненный цикл** — clean, default (validate → … → deploy), site; фазы выполняются по порядку.
- **Плагины** — цели (goals), привязанные к фазам; выполняют компиляцию, тесты, упаковку.
- **Репозитории** — локальный (~/.m2/repository), удалённые, Maven Central.
- **Scope** — compile, provided, runtime, test; от этого зависит classpath и попадание в артефакт.
- **Транзитивность** — зависимости зависимостей подтягиваются автоматически; дерево: `mvn dependency:tree`.
- **dependencyManagement** — централизованное объявление версий (и scope) без автоматического добавления зависимости.

### Ключевые концепции Gradle

- **Build script** — build.gradle (Groovy) или build.gradle.kts (Kotlin).
- **Tasks** — атомарные единицы работы; граф зависимостей между задачами определяет порядок выполнения.
- **Плагины** — добавляют задачи и конфигурации (например, java, application, spring-boot).
- **Конфигурации зависимостей** — implementation, api, compileOnly, runtimeOnly, testImplementation.
- **Wrapper** — gradlew / gradlew.bat + gradle/wrapper; фиксированная версия Gradle для всех.
- **Daemon** — долгоживущий процесс JVM для ускорения сборок.
- **Инкрементальная сборка** — задачи с inputs/outputs пропускаются, если ничего не изменилось (UP-TO-DATE).
- **settings.gradle** — корень сборки и список модулей (include).

### Ключевые различия Maven и Gradle

| Тема | Maven | Gradle |
|------|--------|--------|
| Конфиг | XML, pom.xml | DSL, build.gradle / .kts |
| Модель | Фазы жизненного цикла | Задачи (task graph) |
| Зависимости | scope (compile, provided, test…) | конфигурации (implementation, api…) |
| Скорость | Обычно медленнее | Быстрее (daemon, кэш, инкрементальность) |
| Гибкость | Через плагины и профили | Плагины + произвольный код в скрипте |
| Версия сборки | Установленный Maven | Фиксируется Wrapper (gradlew) |
